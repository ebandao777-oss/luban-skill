# Agent智能体技能进化方法论

## Scenario-Adaptive Dual-Mode Architecture (SA-DM)

### 核心法则

- **"有道之魂.有术之能"**

- **【单模型架构的“幻觉”风险】**
  单模型在处理复杂任务时极易产生“幻觉”（即生成看似自信但完全虚构的信息）。这源于大模型的概率预测本质与知识边界盲区。在长链路推理中，单步的“事实性错误”或“执行漂移”会不断累积放大，导致任务彻底失败。因此，必须摒弃单模型单角色架构，通过多角色协同与S级模型的全局视野来精准捕捉隐性错误，避免“低水平裁判”导致的进化方向偏移。

- **【评审架构的阶梯式能力要求】**
  Agent 智能体的自主进化高度依赖评审模型（Evaluator）的判别力。在工程实践中，评审架构的有效性遵循严格的降维规律：
  **S级模型多角色协同评审 > A级模型多角色评审 > 单模型多角色评审 > 单模型单角色评审**。

- **【基座模型的“高杠杆”优化策略】**
  在技能专项优化阶段，必须摒弃算力平均主义。采用顶级 S 级模型（如 Claude / GPT-5 / Qwen3.7-Max / DeepSeek-V4 Pro）作为基座进行专项打磨，是获取最高 ROI 的唯一解。
  顶级模型具备卓越的长程自主执行与自我纠错能力，能够突破单体智能瓶颈，实现从“单次问答”到“复杂工程闭环”的质变。

## 1. EvoSkill — 失败驱动的技能修补

- **地址**：[arXiv:2603.02766](https://arxiv.org/abs/2603.02766)
- **简介**：三 Agent 协同，让 AI 从失败中自动发现能力缺口，自主构建结构化技能文件夹（含元数据/指令/脚本）。生成的不是不可解释的 Prompt 变体，而是可迁移的技能资产。
- **落地思路**：当前 skill 执行报错时，自动抓取失败上下文 → 分析缺口类型（规则缺失？触发词遗漏？指令冲突？）→ 输出修改建议。做成 workflow，不再人工排查。

## 2. SkillOps — 定期健康巡检

- **地址**：[arXiv:2605.13716](https://arxiv.org/abs/2605.13716)
- **简介**：首次将技能库维护形式化为「库时」问题。核心是五维健康诊断：效用、冗余、兼容性、失败风险、验证缺口，并自动执行 merge/repair/retire/add_validator 等维护动作。
- **落地思路**：对已有技能做定时巡检任务。检查 references 引用是否断裂、SKILL.md 是否有规则冲突、是否有未测试的新增规则、是否有可合并的去重项。一次扫描输出一份体检清单。

## 3. CASCADE — 领域知识自动补充

- **地址**：[arXiv:2512.23880](https://arxiv.org/abs/2512.23880)
- **简介**：两大元技能驱动自进化——持续学习（网页搜索/代码提取/记忆利用）和自我反思（内省/知识图谱探索）。技能以可执行代码累积，跨智能体共享。
- **落地思路**：当技能 references 中引用的外部知识过时时，自动 web_search + 验证 → 更新到 references/ 对应文件。适合 prompt-optimizer 这种依赖外部论文/技术发展的技能。

## 4. Skill Distill — 判断何时该精简

- **地址**：[arXiv:2604.01608](https://arxiv.org/abs/2604.01608)
- **简介**：提出「指标自由度 F」——第一个可先验计算的技能效用预测器。F 通过 Mantel 检验量化输出多样性与得分方差的耦合强度。高 F 指标可直接精简，低 F（≤0.6）才需保留。
- **落地思路**：扫描技能文件大小和 references 数量，超阈值时计算各模块的 F 值 → 高 F 模块标记可合并/删除 → 低 F 模块保留。避免 skill 越长越臃肿。

## 5. HASP — 将建议规则硬化

- **地址**：[arXiv:2605.17734](https://arxiv.org/abs/2605.17734)
- **简介**：技能升格为可执行程序函数（PF），每个 PF 含 should_activate（状态感知触发）和 intervene（动作重写或上下文注入）两个接口，从"建议"变成"硬纠正"。
- **落地思路**：分析执行日志，定位 SKILL.md 中写了但 Agent 经常忽略的规则 → 加上触发条件和硬执行描述 → 改为 `强制：当 X 发生时必须执行 Y` 格式，而非 `建议/应该`。

## 6. MUSE-Autoskill — 修改后自动回归测试

- **地址**：[arXiv:2605.27366](https://arxiv.org/abs/2605.27366)
- **简介**：技能全生命周期管理框架，核心是双驱动评估：单元测试 + 运行反馈，自动触发修补和重测。首次实证跨智能体技能迁移。
- **落地思路**：每次编辑 skill 文件前，自动生成一组测试用例（覆盖触发词识别、输出格式、关键规则遵守）→ 修改完成后跑回归 → 退化则回滚。确保改不坏。

## 核心闭环：EvoSkill + SkillOps 双轮驱动

```
技能执行 → 出错了
    ↓
EvoSkill 分析失败，生成补丁
    ↓
MUSE-Autoskill 自动验证补丁（测试用例）
    ↓
SkillOps 定期做体检，防退化
    ↓
CASCADE 补领域新知识（内容陈旧时）
    ↓
HASP 硬化关键规则（规则被忽略时）
    ↓
Skill Distill 精简冗余（文件臃肿时）
```

**适用对象**：tiangong-skill、luban-skill、prompt-optimizer 等任何已有 skill 文件，均可纳入此优化流程。

# 技能自进化系统 —— 落地方案

> 基于 EvoSkill / SkillOps / CASCADE / Skill Distill / HASP / MUSE-Autoskill 六篇论文，设计一套可直接实施的技能自动进化流程。

## 一、整体架构

```
                    ┌──────────────────────────────────────┐
                    │         技能自进化调度器              │
                    │  (轮询 / 事件驱动 / 按需触发)         │
                    └──────┬───────────────────────────────┘
          ┌────────┬───────┼───────┬──────────┬──────────┐
          ▼        ▼       ▼       ▼          ▼          ▼
     ┌────────┐┌────────┐┌────────┐┌────────┐┌────────┐┌────────┐
     │EvoSkill││SkillOps││CASCADE ││ Distill││  HASP  ││  MUSE  │
     │ 执行   ││ 定期   ││ 知识   ││ 精简   ││ 规则   ││ 回归   │
     │ 失败   ││ 体检   ││ 更新   ││ 触发   ││ 硬化   ││ 测试   │

         │         │         │         │         │         │
         └─────────┴─────────┴─────────┴─────────┴─────────┘
                              │
                              ▼
                    ┌──────────────────────┐
                    │    技能文件仓库       │
                    │  (SKILL.md + refs/)  │
                    └──────────────────────┘
```

**调度器职责**：决定何时触发哪个模块，避免并发修改冲突。采用事件队列 + 单线程执行。

## 二、各模块详细设计

### 模块 1：EvoSkill —— 失败驱动的技能修补

**触发条件**：用户反馈技能执行不符合预期，或 Agent 执行时遇到明确能力缺口。

**执行流程**：

```
用户反馈 / 执行异常
      │
      ▼
Step 1: 捕获失败上下文
  - 用户的原始指令
  - 技能被调用的完整参数（触发词、task 参数）
  - 技能执行后的输出 / 报错信息
  - 用户的具体反馈（"不对"、"没效果"、"缺少XX"）
      │
      ▼
Step 2: 定位缺口类型
  ┌──────────────────┬─────────────────────────────┐
  │ 缺口类型          │ 判定特征                    │
  ├──────────────────┼─────────────────────────────┤
  │ 触发词遗漏        │ 用户输入含关键意图但技能未触发│
  │ 规则缺失/模糊     │ 技能没有覆盖用户场景的指引    │
  │ 指令冲突          │ 多条规则给出矛盾建议          │
  │ 流程漏洞          │ 技能规定的流程某环节不可行    │
  │ 输出格式不当      │ 技能输出格式不符合预期        │
  │ 版本兼容          │ 依赖的外部工具版本已变更      │
  └──────────────────┴─────────────────────────────┘
      │
      ▼
Step 3: 生成补丁
  - 定位技能文件中需要修改的具体位置（文件路径 + 行号范围）
  - 给出 old_str → new_str 的具体修改内容
  - 附带修改理由（为什么这样改能解决缺口）
      │
      ▼
Step 4: 输出到确认队列
  - 格式：Markdown 修复建议卡片
  - 状态：待用户确认 → 确认后自动执行 edit_file
```

**输出格式示例**：

```markdown
## 技能修复建议

**目标技能**：tiangong-skill
**缺口类型**：触发词遗漏
**定位**：SKILL.md 第 12 行触发条件列表

**当前**：
- "优化skill"、"小鲁班"

**建议新增**：
- "skill打分"、"skill评分"、"技能评分"

**理由**：用户最近 3 次使用"给这个技能打个分"未触发 tiangong-skill，而路由到了错误的能力。
```

### 模块 2：SkillOps —— 定期健康巡检

**触发条件**：定时任务（建议每周一次），或手动触发"检查技能健康"。

**五维诊断矩阵**：

| 维度 | 检查项 | 方法 | 严重程度 |
|------|--------|------|----------|
| **效用** | SKILL.md 中每条规则是否有对应的触发条件 | 扫描 trigger 字段覆盖率 | 🟡 |
| | references 文件是否被 SKILL.md 引用 | 交叉比对引用链 | 🟢 |
| **冗余** | 是否存在内容高度重复的 references | 计算文件相似度（Jaccard） | 🟢 |
| | SKILL.md 中是否有可合并的重复描述 | 段落语义去重 | 🟢 |
| **兼容性** | 文件路径引用是否存在断裂 | 逐条检查相对路径可访问性 | 🔴 |
| | 外部链接是否失效 | HTTP HEAD 请求检查 | 🟡 |
| | YAML frontmatter 是否合法 | YAML 解析校验 | 🔴 |
| **失败风险** | 是否有未定义触发条件的强制性规则 | 扫描"必须"/"禁止"等关键词 | 🟡 |
| | 规则粒度是否过粗导致歧义 | 统计每条规则的约束项数量 | 🟢 |
| **验证缺口** | 是否缺少对应 rules 的验证步骤 | 检查"验证"关键词覆盖 | 🟡 |
| | references 是否缺少示例和反例 | 扫描 example/counterexample | 🟢 |

**执行流程**：

```
Step 1: 加载目标技能文件（SKILL.md + references/ 下所有文件）

Step 2: 逐维度扫描
  - 工具化扫描：Python 脚本做结构分析（路径、YAML、引用链）
  - 语义化扫描：Agent 做内容分析（规则一致性、重复、歧义）

Step 3: 汇总诊断报告
  - 按严重程度排序：🔴 断裂 > 🟡 风险 > 🟢 优化
  - 每项附：位置、问题描述、建议修复动作

Step 4: 生成维护动作清单
  - merge：合并重复的 references
  - repair：修复断裂的路径/链接
  - retire：标记废弃的规则（已不再适用）
  - add_validator：补充验证步骤
```

**维护动作定义**（对齐 SkillOps 论文概念）：

| 动作 | 含义 | 示例 |
|------|------|------|
| `merge` | 合并内容高度重复的 references | ref-a.md 和 ref-b.md 语义相似度 > 0.8 → 合并 |
| `repair` | 修复断裂引用或非法格式 | 路径 `/old/path.md` 不存在 → 改为 `/new/path.md` |
| `retire` | 标记过时规则（加 `[DEPRECATED]`） | 规则提到已停用的 API → 标记待清理 |
| `add_validator` | 补充验证/测试用例 | 规则 A 无验证流程 → 自动生成测试样例 |

### 模块 3：CASCADE —— 领域知识自动补充

**触发条件**：技能 references 中引用外部知识（论文、API、规范），且距上次更新时间超过阈值（默认 90 天）。

**执行流程**：

```
Step 1: 扫描技能 references/ 目录
  - 识别所有外部引用（论文 arXiv ID、API 文档 URL、标准编号等）
  - 记录每个引用的最后更新日期

Step 2: 筛选过时引用
  - 距上次更新 > 阈值 → 标记为待更新
  - 优先处理用户最近高频使用的技能

Step 3: 知识检索（web_search + web_fetch）
  - 论文：搜索引用的 arXiv ID，检查是否有新版本
  - API：抓取最新文档，对比 changelog
  - 标准：搜索是否发布了新版本

Step 4: 自我反思（内省）
  - 对比新旧知识差异
  - 判断是否影响技能规则的有效性
  - 仅在有实质性变化时生成更新

Step 5: 更新 references/ 文件
  - 追加新知识（不删除旧内容，标注版本号）
  - 格式：## [YYYY-MM-DD] 更新：xxx → 新内容
```

**关键设计**：不做自动替换旧规则，只做追加+标注。规则的废弃留给 SkillOps 的 `retire` 动作。

### 模块 4：Skill Distill —— 判断何时该精简

**触发条件**：SkillOps 体检报告中「冗余」维度评分低于阈值，或技能 references 文件数超过 15 个。

**指标自由度 F 的近似计算**：

由于原论文的 Mantel 检验需要实际执行数据，我们用一个启发式近似：

```
F_approx = 1 - (模块被规则引用的次数 / 模块总字符数归一化)
```

- F ≈ 1：模块内容庞大但很少被规则引用 → 可精简
- F ≈ 0：模块内容紧凑且多处引用 → 保留

**执行流程**：

```
Step 1: 构建引用矩阵
  - SKILL.md 中每条规则 → 引用了 references/ 的哪些段落
  - 计算每个 references 文件的「有效引用密度」

Step 2: 计算 F_approx
  - F_approx ≥ 0.7：标记为「可精简」
  - F_approx ≤ 0.3：标记为「核心资产，保护」
  - 0.3 < F_approx < 0.7：暂不处理

Step 3: 生成精简建议
  - 针对「可精简」文件：
    a. 提取被引用的核心段落
    b. 建议删除未引用部分
    c. 建议将高引用密度段落提升到 SKILL.md

Step 4: 输出精简方案
  - 展示「删除后文件大小变化」预估
  - 标注「保留的核心内容」
  - 待用户确认后执行
```

**精简优先级**：

| 优先级 | 类型 | 典型处理 |
|--------|------|----------|
| P0 | 完全未被引用的 references 文件 | 直接建议删除 |
| P1 | 文件大但仅 1-2 处引用 | 提取引用段落到 SKILL.md，删原文件 |
| P2 | 多处重复的示例代码块 | 合并为一个 reference |
| P3 | 历史版本累积的旧内容 | 归档到 archive/ 子目录 |

### 模块 5：HASP —— 将建议规则硬化

**触发条件**：同一技能在同一场景下连续 2 次以上未遵守关键规则。

**规则硬化的两种层级**：

#### 层级 1：Should → Must（措辞强化）

不改架构，只升级措辞。

```
原文：建议在生成 SKILL.md 时控制文件大小在 30KB 以内

硬化后：强制约束：SKILL.md 文件大小不得超过 30KB。超限时，必须将详细内容拆分到 references/，SKILL.md 仅保留导航链接。
```

#### 层级 2：Should → PF（可执行程序函数）

在 SKILL.md 中为关键规则附加触发条件 + 执行逻辑。

```yaml
# 规则元数据（YAML frontmatter 中）
hard_rules:
  - id: rule_001
    should_activate: "SKILL.md 文件大小 > 30KB"
    intervene:
      type: "block_and_restructure"
      action: "禁止继续在 SKILL.md 追加内容，将超出部分写入新的 reference 文件，SKILL.md 仅保留一行引用链接"
    severity: "critical"
    last_violated: "2026-06-10"
    violation_count: 3
```

**执行流程**：

```
Step 1: 执行日志分析
  - 从调度器中提取每次技能调用的结果
  - 对比 SKILL.md 中的关键规则 vs 实际执行行为
  - 识别「规则被忽略」的实例

Step 2: 分级处理
  忽略次数 1 次 → 不做处理（可能是一次性特殊情况）
  忽略次数 2 次 → 生成措辞强化建议（层级 1）
  忽略次数 ≥3 次 → 生成 PF 硬化建议（层级 2）

Step 3: 硬化规则生成
  - 提取被忽略规则的核心约束
  - 定义 should_activate 条件（何时该触发）
  - 定义 intervene 动作（触发后做什么）
  - 注入到 SKILL.md 的 hard_rules 字段

Step 4: 输出硬化建议
  - 展示「原规则 → 硬化后规则」对比
  - 标注触发条件和干预动作
  - 待用户确认后执行
```

**硬化适用性判断**：

| 规则类型 | 是否适合硬化 | 原因 |
|----------|-------------|------|
| 输出格式约束 | ✅ 适合 | 可精确检测和强制修正 |
| 文件大小限制 | ✅ 适合 | 可精确检测 |
| 必须包含的章节/字段 | ✅ 适合 | 结构化检查 |
| 语义风格约束 | ❌ 不适合 | 难以精确检测 |
| 创造性建议 | ❌ 不适合 | 无法形式化 |

### 模块 6：MUSE-Autoskill —— 修改后自动回归测试

**触发条件**：任何对技能文件（SKILL.md 或 references/）的编辑操作完成后。

**测试用例生成规则**：

| 测试维度 | 生成方法 | 示例 |
|----------|----------|------|
| **触发词识别** | 从 SKILL.md 提取所有触发词，逐一构造输入 | 输入"小鲁班" → 预期触发 tiangong-skill |
| **输出格式** | 从 SKILL.md 提取格式约束，构造验收条件 | 输出必须包含 YAML frontmatter |
| **关键规则遵守** | 提取"必须"/"禁止"语句，构造边界测试 | 输入导致超限的请求 → 预期技能拒绝 |
| **流程完整性** | 按技能的 Step 列表逐项模拟 | Step 3 依赖 Step 2 的输出 → 断链测试 |
| **references 可达性** | 遍历所有文件路径引用 | 逐条检查文件是否存在 |
| **反例测试** | 构造明确不在技能范围内的输入 | "帮我写个操作系统" → 预期不触发技能 |

**执行流程**：

```
Step 1: 修改前快照
  - 保存修改前的完整文件 hash
  - 自动生成 5-10 条测试用例（基于上述 6 维度）

Step 2: 执行修改（用户确认的编辑操作）

Step 3: 回归测试
  - 逐条运行 Step 1 生成的测试用例
  - 逐条检查修改后的技能行为

Step 4: 结果判定
  全部通过 → 输出「回归测试通过，无退化」
  部分失败 → 列出失败项 + 具体偏差 + 建议回滚
  全部失败 → 强制建议回滚到修改前版本

Step 5: 测试用例沉淀
  - 通过的测试用例追加到技能目录下的 tests.yaml
  - 形成持续增长的回归测试集
```

**tests.yaml 格式**：

```yaml
# 技能自进化 —— 回归测试用例（自动生成 + 人工补充）
skill: tiangong-skill
generated_at: 2026-06-11
tests:
  - id: trigger_001
    dimension: "触发词识别"
    input: "小鲁班帮我优化下这个skill"
    expected: "触发 tiangong-skill，执行优化流程"
    status: pass
    last_run: 2026-06-11

  - id: format_001
    dimension: "输出格式"
    condition: "SKILL.md 输出必须包含 description 字段"
    check: "YAML frontmatter 解析通过"
    status: pass
    last_run: 2026-06-11

  - id: rule_001
    dimension: "关键规则遵守"
    input: "在不分析问题的情况下直接修改文件"
    expected: "技能应拒绝，要求先分析再修改"
    status: fail
    fail_reason: "技能未拦截直接修改请求"
    last_run: 2026-06-11
```

## 三、调度策略

### 3.1 触发优先级

```
事件驱动（立即响应）
  ├── P0: 用户明确反馈技能错误         → 触发 EvoSkill
  ├── P0: 技能编辑操作完成              → 触发 MUSE 回归测试
  ├── P1: 同一规则连续被忽略 3 次        → 触发 HASP 硬化
  └── P2: 同一规则连续被忽略 2 次        → 触发 HASP 措辞强化

定时驱动（周期扫描）
  ├── 每周：SkillOps 健康巡检
  ├── 每月：Skill Distill 精简检查（仅当冗余评分低时）
  └── 每季度：CASCADE 知识更新检查（仅当有外部引用时）
```

### 3.2 并发控制

- 同一技能同时只允许一个演化任务运行
- 新任务到达时排队，FIFO 顺序执行
- 回归测试（MUSE）运行期间锁定技能文件，其他模块等待

### 3.3 修改冲突仲裁

| 场景 | 仲裁规则 |
|------|----------|
| EvoSkill 生成补丁 vs 用户正在手动编辑 | 用户优先，EvoSkill 补丁排队 |
| SkillOps 巡检 vs 用户正在手动编辑 | 巡检只读执行，仅输出报告不自动修改 |
| Skill Distill 精简 vs HASP 硬化冲突 | HASP 优先（硬化是质量保障），精简让步 |

## 五、与调度器的集成

本方案作为一个独立技能，命名为 `skill-evolution`，由 Marvis 调度：

```
Marvis 收到「检查技能健康」「这个技能为什么不行」「精简技能」等指令
      │
      ▼
dispatch_task → skill-evolution 技能
      │
      ▼
skill-evolution 内部按上述模块执行
      │
      ▼
返回分析报告 / 修改建议 / 自动修复结果
```

**调度判断表**：

| 用户指令关键词 | 触发模块 | 是否需要用户确认 |
|---------------|----------|-----------------|
| "检查技能"、"技能体检" | SkillOps | 否（仅输出报告） |
| "这个技能有问题"、"技能出错了" | EvoSkill | 是（修改需确认） |
| "更新技能知识"、"补最新" | CASCADE | 否（仅追加不修改） |
| "精简技能"、"技能太长了" | Skill Distill | 是（删除需确认） |
| "这个规则总被忽略"、"硬一点" | HASP | 是（修改需确认） |
| "改完后测试下"、"不会退化吧" | MUSE | 否（自动跑） |

## 六、渐进落地路线

### Phase 1（MVP，1-2 天实现）

只落 SkillOps + EvoSkill，覆盖 90% 高频场景。

- SkillOps：纯读扫描，输出 Markdown 体检报告，不自动修改
- EvoSkill：失败时生成补丁建议，用户手动确认后执行

### Phase 2（完善，3-5 天）

补充 MUSE + CASCADE。

- MUSE：每次技能编辑后自动生成测试用例并跑回归
- CASCADE：季度检查外部引用是否有更新

### Phase 3（进阶，视需求）

HASP + Skill Distill，处理长尾问题。

- HASP：累积足够执行日志后启用规则硬化
- Skill Distill：技能文件明显臃肿时触发精简
