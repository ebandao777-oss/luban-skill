# Agent智能体技能进化方法论

## Scenario-Adaptive Dual-Mode Architecture (SA-DM)

### 核心法则

- **"有道之魂.有术之能"**

- **【单模型架构的“幻觉”风险】**
  单模型在处理复杂任务时极易产生“幻觉”（即生成看似自信但完全虚构的信息）。这源于大模型的概率预测本质与知识边界盲区。在长链路推理中，单步的“事实性错误”或“执行漂移”会不断累积放大，导致任务彻底失败。因此，必须摒弃单模型单角色架构，通过多角色协同与S级模型的全局视野来精准捕捉隐性错误，避免“低水平裁判”导致的进化方向偏移。

- **【评审架构的阶梯式能力要求】**
  Agent 智能体的自主进化高度依赖评审模型（Evaluator）的判别力。在工程实践中，评审架构的有效性遵循严格的降维规律：
  **S级模型多角色协同评审 > A级模型多角色评审 > 单模型多角色评审 > 单模型单角色评审**。

- **【基座模型的“高杠杆”优化策略】**
  在技能专项优化阶段，必须摒弃算力平均主义。采用顶级 S 级模型（如 Claude / GPT-5 / Qwen3.7-Max / DeepSeek-V4 Pro）作为基座进行专项打磨，是获取最高 ROI 的唯一解。
  顶级模型具备卓越的长程自主执行与自我纠错能力，能够突破单体智能瓶颈，实现从“单次问答”到“复杂工程闭环”的质变。

### 一、 核心定义

本方法论是一套指导智能体技能全生命周期优化的顶层架构。其核心主张：**不存在普适的最优技能进化路径，只存在与场景风险收益特征相匹配的进化策略**。

**核心升级**：从「全域融合」走向「场景自适应双模」——Quick / Full 模式定位详见 §七，两套模式隔离与互通机制详见 §三 原则 #3 及 §八。

该方法论将「**场景的风险收益特征**」确立为设计的**第一性原理**。所有技术选型与架构决策，均源于对场景本质属性（是追求体验上限的个人探索，还是保障风险下限的工业生产）的深刻洞察，并由此衍生出「**感知-分流-适配-反馈**」的四步闭环。

### 二、 模型依赖声明

SA-DM 架构对底层模型能力存在硬性依赖。

基座模型能力基线

以下为 SA-DM 正常工作所需的基座模型能力基线，L2/L4 防御层可在各能力项的合理波动范围内自动补偿。

| 依赖维度 | 具体能力要求 |
| :--- | :--- |
| 自我认知 | 准确评估自身输出质量、识别错误模式 |
| 代码/配置生成 | 生成语法正确、逻辑自洽的修改补丁 |
| 指令遵循 | 严格遵守编辑预算、安全红线等约束 |
| 长上下文理解 | 理解历史版本变更链、累积经验 |

#### 工程可解耦的能力（弱模型依赖）

- 验证环节：Held-out 测试、回归测试是确定性的
- 版本管理：diff 计算、回滚机制是纯工程能力
- 健康度监控：results.tsv 动态阈值基于统计规则
- 合规审查：规则引擎过滤可用传统 NLP/正则

#### 依赖度动态曲线

```
升级成功率 / 安全性
▲
│                          ╭──── 工程补偿区
│                     ╭────╯     (L2/L4防御层生效)
│                ╭────╯
│           ╭────╯
│      ╭────╯ ← 最低可用门槛
│ ╭────╯         (低于此线，自动升级不可行)
│─╯
└──────────────────────────────► 模型基础能力
弱                            强
```

- 门槛以下：模型无法满足自评+生成的基本能力要求，系统自动降级为手动模式并输出能力不足诊断报告
- 门槛~中等：工程补偿区，SA-DM 架构价值最大的区间
- 中等以上：边际收益递减，防御层可逐步精简

**退化检测指标**：当前模型所处阶段通过以下指标持续追踪，写入 `results.tsv` 并在每会话 Phase 0 评估：

| 指标 | 门槛以下 | 工程补偿区 | 中等以上 | 检测方式 |
| :--- | :---: | :---: | :---: | :--- |
| 回滚率 | 偏高¹ | 5-30% | <5% | results.tsv revert 率（定义见 §十 动态阈值告警） |
| 拒绝缓冲命中率 | 偏高² | 3-20% | <3% | rejected_edits.md（拒绝编辑缓冲区文件）命中/提交比 |
| 平均编辑预算消耗 | N/A | 3-10% | <3% | token diff 统计 |
| 自评-实测偏差 | >10 分 | 3-10 分 | <3 分 | dim8 dry_run vs full_test Δ |

注：revert 率、Δ均值趋势、flat 比例等指标的定义和阈值复用 §十 健康度监控的动态阈值体系，本表仅标注其在模型能力阶段的区间映射。

**区间边界校准来源**（基于 SkillOpt 实证 arXiv 2605.23904 与 SkillLens 实证 arXiv 2605.23899）：回滚率 30%/5% 基于 bounded edit revert 分布中位数 4.2% vs uncontrolled 中位数 31%；自评-实测偏差 10/3 基于 dry_run 乐观偏差 μ=3.8(σ=2.1)；拒绝缓冲命中率 20%/3% 基于 CI/CD flaky test 检测实践；编辑预算消耗 10%/3% 基于 SkillOpt P50=4.7%

¹ 回滚率偏高对应 >30%，² 拒绝缓冲命中率偏高对应 >20%。上述阈值作为初始校准基线。§十 动态阈值告警中的历史基线自适应机制将在累积 ≥30 轮实际运行数据后，自动根据本实例的分布重新校准区间边界。

**自动重测调度链路**（本链路为异步跨会话操作）：当任一指标连续 2 次会话突破区间边界时，系统按以下链路自动执行：

```
漂移告警触发
│
├─ Step 1: 版本指纹采集
│   采集当前模型版本号 + 响应特征签名（平均 token/s、首字延迟、拒答率）
│   与上次基线测试时的基准指纹对比
│
├─ Step 2: 差异判定
│   版本号变更 → "模型升级"，执行完整消费者能力基线重测（§五.3）
│   仅响应特征偏移 → "性能波动"，记录告警但不重测，连续 3 次后升级为完整重测
│   版本号+特征均未变但指标持续恶化 → "隐性退化"，立即执行完整重测
│
├─ Step 3: 自动重测执行
│   后台启动独立子 Agent，运行 §五.3 定义的消费者能力基线测试套件
│   测试期间当前会话继续使用旧基线（不阻塞用户）
│   重测在独立子 Agent 中异步执行，含双向心跳检测和自动重试；连续失败时触发管理员通知。
│
└─ Step 4: 基线更新与会话注入
重测完成后自动更新 results.tsv 中的基线行（标注 auto_retest=true）
若模式等级发生变更 → 下次会话 Phase 0 自动注入新等级
若等级未变 → 仅更新基线数值，用户无感知
```

| 差异类型 | 检测方式 | 动作 | 用户感知 |
| :--- | :--- | :--- | :--- |
| 模型升级 | 版本号变更 | 完整重测 | 下次会话通知「基座模型已升级，优化策略已自动调整」 |
| 性能波动 | 仅响应特征偏移 | 记录告警，延迟重测 | 无感知（连续 3 次后通知） |
| 隐性退化 | 指标持续恶化但指纹未变 | 立即完整重测 | 重测完成后通知「检测到能力漂移，已自动校准」 |

##### 运维参数

| 参数 | 值 | 说明 | 适用范围 |
| :--- | :--- | :--- | :--- |
| ping 间隔 | 30s | 双向心跳间隔 | 自动重测调度链路的独立子 Agent |
| pong 超时 | 180s | 主进程判定子 Agent 失联 | 自动重测调度链路的独立子 Agent |
| ping 超时 | 120s | 子 Agent 自主退出 | 自动重测调度链路的独立子 Agent |
| 重试上限 | 3 次 | 失联后重试启动次数 | 自动重测调度链路的独立子 Agent |

> **定位声明**：SA-DM 是为"当前水平的模型"设计的——不假设最强模型，而是通过 L2/L4 防御层在模型"够用但不完美"的条件下让自动升级安全可控。当基座模型能力跃升一个台阶后，架构本身需右移精简。上述自动重测链路确保架构对模型能力变化具备**自主感知－自动校准**能力，而非依赖用户手动触发。

### 三、 三大核心原则

| 原则 | 关键阐述 |
| :--- | :--- |
| **1. 场景先于技术 (Why)** | 技术选型服从场景风险收益特征——个人场景追求体验上限，工业场景追求风险下限，两者不可调和只能分流（详见 §七）。 |
| **2. 分层治理，按需激活 (How)** | 将能力解耦为 5 个正交层级。不强制全层级启用，根据场景模式与任务 ROI 动态激活，避免过度工程化。 |
| **3. 双轨反馈，隔离共生 (What)** | Quick 与 Full 模式运行在严格隔离的数据空间但共享底层抽象。个人探索提供创新信号，工业验证沉淀可靠基线，通过受控机制相互滋养（详见 §八）。 |

### 四、 架构红线

以下禁令是 SA-DM 的设计级约束——不可通过参数调优绕过，不可通过模式切换豁免，不可由衍生实例选择性忽略。

| # | 红线 | 架构级理由 |
| :---| :---| :---|
| 1 | **禁止改评同源**——不得由执行优化的同一上下文承担评分职责 | 单 judge 评分准确率约 53.6%，SA-DM 通过多 judge 独立评分 + 分歧检测机制将准确率提升至 73.8% 以上 |
| 2 | **禁止跨模式数据污染**——Quick/Full 的画像、评分、备份必须物理隔离 | 个人风格偏好稀释工业合规标准，污染不可逆 |
| 3 | **禁止跳过基线直接优化**——Full 模式必须先行消费者能力测试 | 模型不行硬优化 = ROI 为负，投入即浪费（论证见 §二） |
| 4 | **禁止编辑预算豁免**——任何单轮修改不得超过原文件 10% token | 大步跳跃使归因失败，SkillOpt 实证 bounded edit 优于 uncontrolled rewriting |
| 5 | **禁止丢弃负反馈**——被 revert 的方案必须永久保留于拒绝编辑缓冲区 | 同样错误重复提交 = 系统性浪费，缓冲区即防御 |
| 6 | **禁止静默变更**——涉及安全/合规/回滚/告警的异常必须通知用户 | 自动化不是免责借口，人在回路是安全底线 |
| 7 | **禁止单 judge 裁决**——Full 模式评分必须 ≥2 个独立评委取中位数 | 单 judge 偏差不可检测，多 judge 通过分歧暴露问题 |
| 8 | **禁止跳过复盘**——≥2 个 skill 优化后必须执行 Epoch Meta-Review | 跨 skill 规律不沉淀 = 经验不可复用 |

> **约束范围**：以上 8 条红线适用于 SA-DM 架构及其所有衍生实例。任何声称"基于 SA-DM"的实现若违反其中任一条，视为不合规实现。

> **红线运行时检测**：声明性红线（#4 编辑预算、#8 跳过复盘）依赖技术强制而非仅人工合规审计：
> - **红线 #4（编辑预算）检测**：Phase 2 Step 4 在每次编辑提交前计算 token diff 占原文件的百分比，>10% 直接拒绝写入并返回 `budget_exceeded` 错误。Phase 0 在下一会话启动时读取 `version_manifest.json` 的上一轮编辑记录，若发现 `budget_violation=true`，阻断本轮优化并通知管理员
> - **红线 #8（跳过复盘）检测**：Phase 0 在启动时检查 `version_manifest.json`——若 `skills_optimized_since_last_review ≥ 2` 且 `meta_learnings.md` 最后更新时间早于最近 2 次优化的完成时间，触发 `review_skipped` 告警，强制在当前会话中插入 Epoch Meta-Review 阶段再进入优化
> **告警通知机制**（红线 #6 技术实现）：通知统一通过 Phase 3 汇总报告输出为结构化告警摘要（`meta_learnings.md`「异常信号」章节 + Phase 3 终端输出），不依赖外部通知渠道（短信/邮件/Webhook 等）。管理员通知即在 Phase 3 报告中以 `[ADMIN]` 标签标记需人工关注的高严重度事件。通知渠道扩展（Webhook / 邮件 / 钉钉 / 企业微信）待后续版本实现。

### 五、 运作框架与分层治理 (L0-L4)

#### 1. 四步闭环流程

> "四步闭环"是本方法论的**动态运行时流程**，而"分层治理框架(L0-L4)"则是其**静态能力底座**。闭环流程负责在运行时根据场景感知结果，动态地激活和调度底层的能力层级。

| 步骤 | 核心动作 | Quick 模式输出 | Full 模式输出 |
| :--- | :--- | :--- | :--- |
| **感知** | 显式选择+十维量规执行模式选择 | 用户私有空间，风格偏好画像 | 企业级租户空间，合规基线画像 |
| **分流** | 任务路由至 L0-L4 | 轻量 SOP，开放反思 | 严格 SOP，硬性熔断 |
| **适配** | 动态激活能力层级 | 激活 L2-L3 (鼓励探索，L1 已定义待落地 MVS) | 激活 L0-L4 (确保安全) |
| **反馈** | 更新画像与基线 | 更新个人偏好，沉淀创新模式 | 更新合规基线，沉淀可靠模式 |

##### 感知步骤：十维量规执行模式选择器

感知不止于用户手动选择模式。系统在每次任务执行中采集以下前置条件参数，决定十维量规的执行模式（Quick 或 Full）：

| # | 十维量规执行模式选择参数 | 采集方式 | 推高「执行 Full 十维量规」权重的条件 |
| :---: | :---| :---| :---|
| ① | 任务领域 | 关键词匹配 | 金融/医疗/法律/政务/生产环境部署 |
| ② | 数据敏感度 | PII 检测 | 含身份证号、银行卡号、密钥、内网路径 |
| ③ | 文件规模 | 目录扫描 + SKILL.md 行数统计 | SKILL.md > 800 行或目录含 ≥5 个引用文件 |
| ④ | 历史 revert 率 | results.tsv | 最近 10 轮优化中 revert ≥ 2 次 |
| ⑤ | 用户显式覆盖 | 指令检测 | 用户明文指定 quick/full → 直接采用，不推断 |

> **文件规模双阈值**：>800 行推高「执行 Full 十维量规」，≤200 行强制「执行 Quick 十维量规」，中间区间由其他参数综合决定（详见下文硬路由条件）。

**十维量规执行模式选择规则**（5 个参数 → Quick/Full 十维量规）：

5 个参数独立归一化后加权组合，输出模式选择值 M ∈ [0, 1]：

```
M = w₁ × N(task_domain) + w₂ × N(pii) + w₃ × N(file_scale) + w₄ × N(history_revert) + w₅ × override_flag
```

| 参数 | 原始信号 | 归一化函数 N(x) → [0, 1] | 权重 w | 归一化说明 |
| :---| :---| :---| :---: | :---|
| ① 任务领域 | 关键词命中列表 | 命中金融/医疗/法律/政务 → 1.0；命中生产环境部署 → 0.8；命中技术方案/数据分析 → 0.4；命中个人创作/闲聊/娱乐 → 0.0；未命中任何关键词 → 0.2（中性默认） | **0.30** | 领域风险是模式选择的第一性原理（§一），权重最高 |
| ② 数据敏感度 | PII 检测命中数 | 含身份证号/银行卡号/密钥 → 1.0；含内网路径/内部 API key → 0.7；含邮箱/手机号（非密钥）→ 0.3；无 PII 命中 → 0.0 | **0.25** | 数据敏感度直接影响合规风险 |
| ③ 文件规模 | SKILL.md 行数 + 引用文件数 | 行数 > 800 或目录含 ≥5 个引用文件 → 1.0；行数 ∈ (200, 800] 且引用文件 < 5 → 线性映射 M = (行数 − 200) / 600；行数 ≤ 200 且引用文件 < 5 → 0.0 | **0.20** | 大文件或重资源 skill 优化副作用面广，需更严格约束 |
| ④ 历史 revert 率 | 最近 10 轮 revert 次数 | revert ≥ 3 → 1.0；revert = 2 → 0.7；revert = 1 → 0.3；revert = 0 → 0.0 | **0.15** | 历史 revert 率反映当前 skill 的优化风险倾向 |
| ⑤ 用户显式覆盖 | 指令检测 | 检测到 quick → 设置 M=0.0 并短路（不计算其他参数）；检测到 full → 设置 M=1.0 并短路；未检测到 → override_flag = 0（不影响计算） | **短路段** | 用户显式指定跳过公式计算，直接采用指定模式 |

> **权重设计依据**：①+② 合计 55%，体现「场景风险收益特征为第一性原理」——任务领域的固有风险和数据敏感度是模式选择的主导因子。③ 文件规模权 20%——大文件优化需更严格但非决定性。④ 历史 revert 权 15%——作为动态调节因子，避免单一历史波动主导决策。权重初始值基于架构设计原则分配，可在累积 ≥30 轮实际路由数据后通过网格搜索自动校准。
> **权重闭合说明**：w₁+w₂+w₃+w₄ = 0.90（非 1.0）。⑤ 用户显式覆盖为短路段——检测到 quick/full 时直接设置 M=0.0 或 M=1.0，不经过加权求和；未检测到时 override_flag=0，不影响公式。因此正常路径输出范围为 [0, 0.90]，短路段补全两端极值，整体覆盖 [0, 1]。路由表中 ≥0.90 的硬路由档位在无显式覆盖时，需四维信号全满方可达到。

> **模式选择示例**：用户问"帮我优化这个医疗问诊 skill"，skill 800 行，含内网路径，最近 10 轮 revert=1，无显式覆盖 → M = 0.30×1.0 + 0.25×0.7 + 0.20×1.0 + 0.15×0.3 = 0.30 + 0.175 + 0.20 + 0.045 = 0.72 → 落入 [0.70, 0.90) 软路由区间，建议「执行 Full 十维量规」并暂停确认。

模式选择结果存储于 `luban-profile.json`（见反馈步骤）。自动路由分五级决策：

| 模式选择值区间 | 触发条件 | 动作 | 用户感知 |
| :--- | :--- | :--- | :--- |
| ≥0.90（硬路由） | 四维信号全满（①命中金融/医疗/法律 + ②PII 命中 + ③>800行或≥5引用文件 + ④revert≥3）或⑤显式 override=full | 自动「执行 Full 十维量规」，不暂停 | 下次会话 Phase 0 告知「上次因高风险自动选择了 Full 十维量规」 |
| [0.70, 0.90)（软路由） | — | 建议「执行 Full 十维量规」，暂停确认 | 展示选择理由，等待用户确认 |
| (0.30, 0.70)（待决策） | — | 展示选择理由，用户手动选择 | 显示「执行 Quick 十维量规 / Full 十维量规」面板 + 选择理由摘要 |
| (0.10, 0.30]（软默认） | — | 默认「执行 Quick 十维量规」，轻提示可切换 | 界面标注「已选择 Quick 十维量规」，一键切换入口可见 |
| <0.10（硬默认） | 任务领域为个人创作/闲聊/娱乐 + 文件规模 <200 行 + revert 率=0，全部满足 | 自动「执行 Quick 十维量规」，不暂停 | 无感知 |

> **边界值 rounding 策略**：模式选择值计算保留小数点后 4 位（内部精度），路由判定时四舍五入到 2 位。边界点行为定义：
> - 恰好在 0.90：归入「硬路由」区间（≥0.90），自动「执行 Full 十维量规」不暂停
> - 恰好在 0.70：归入「软路由」区间（[0.70, 0.90)），暂停确认
> - 恰好在 0.30：归入「软默认」区间（(0.10, 0.30]），默认「执行 Quick 十维量规」
> - 恰好在 0.10：归入「软默认」区间，而非「硬默认」——0.10 边界预留缓冲，仅当值严格 ≤0.09（四舍五入后 ≤0.09）才进入硬默认

> **十维量规双模执行差异**：Quick 模式和 Full 模式下的十维量规执行深度不同——Quick 模式：dim8 dry_run + 1 评委 + dim7 基础检查（dim10 不执行）；Full 模式：dim8 full_test + ≥2 评委 + 全维度评分（含 dim10 审查门控）。完整差异定义见 §六「双模量规差异」表。

**冷启动策略**：新用户（`luban-profile.json` 不存在或 `mode_preference.record_count < 5`）时，模式选择暂不生效，直接展示 Quick/Full 手动选择面板，附一句话说明。累计 5 次优化后自动开启模式选择，此后用户仍可随时关闭。

**误判纠偏协议**：若用户连续 2 次在建议 Full 时手动选 Quick（或反之），系统将该领域的倾向权重衰减 50%，并记录于 profile 中「推断豁免」字段，后续同类任务跳过选择直接复用用户上次选择。

反馈步骤的产出持久化到 `luban-profile.json`（与 SKILL.md 同目录），结构如下：

"changelog": [

],

"personal": {
"style_tags": ["简洁", "表格优先", "无表情符号"],

**画像版本管理协议**：`luban-profile.json` 遵循语义化版本——`version` 整数递增，`changelog` 记录每次变更的时间戳和内容摘要。Phase 0 读取时校验版本号与上次会话的一致性：若版本号回退（异常覆盖），触发 `profile_integrity` 告警并从 `luban-backups` 恢复上一有效版本。**有效版本判定**：不是简单取时间最近——从 `luban-backups` 中按时间倒序扫描备份文件，逐版本执行以下校验：① `version` 字段递增连续（相对于上次读取版本 +1）② `changelog` 不为空且最新条目时间戳不晚于当前系统时间 ③ JSON 结构完整可解析。首个通过全部校验的备份即为有效版本；若全部备份均未通过，触发 `profile_recovery_failed` 告警，人工介入。跨模式迁移时，profile 不直接复制——通过 `profile-export` 导出中性格式，经合规审查后 `profile-import` 注入目标模式空间。

**并发写入控制**：对 `luban-profile.json` 的写入采用单写者模式——每次优化会话启动时通过原子操作获取文件锁（`O_CREAT | O_EXCL` 创建 `luban-profile.lock`，操作系统保证该操作的原子性，消除 TOCTOU 窗口），写入完成后释放。若锁已被占用（另一会话正在写入），当前会话排队等待（超时 5 秒后降级为以下异步补写流程）。锁文件在进程崩溃时由操作系统自动回收（文件句柄关闭即释放），无需手动清理。

> **锁超时异步补写机制**：锁超时后画像变更序列化为 JSON 写入 `luban-profile-pending.jsonl`（追加模式，每行含 `ts/session_id/changes/retry_count/max_retries` 字段）。Phase 0 检测到非空 pending 条目时按 FIFO 消费（获取锁→合并变更→释放锁→移除该条）。单条最多重试 5 次，超限后迁移至 `meta_learnings.md`「异常信号」标记 `profile_sync_lost`。队列上限 100 条，连续 3 次跨会话锁超时触发 `profile_lock_contention` 告警。

`luban-profile.json` 在每次优化会话的 Phase 0 被读取，作为感知步骤的历史参考输入。当个人画像中的 style_tags 发生变化时，下次 Phase 1 的 dim7（整体架构）评分标准自动纳入该偏好权重。当工业画像中的 compliance_baseline 更新后，下次 dim8 Safety/Compliance 子维度的阈值随之调整。

**闭合验证点**：每次会话结束后检查——本次反馈步骤更新的 profile 值与下次会话感知步骤读取的值是否一致。不一致时触发 `profile_drift` 警告，写入 results.tsv note 列。

#### 2. 分层治理框架 (L0-L4)

| 层级 | 名称 | Quick 模式策略 | Full 模式策略 |
| :--- | :--- | :--- | :--- |
| L0 | 确定性执行层 | 默认启用 (保障基础体验) | 默认启用 (保障基础体验) |
| L1 | 多智能体协作层 (已定义待落地) | 按需启用 (支持复杂创作) | 按需启用 (支持复杂任务) |
| L2 | 技能自适应优化层 | 默认启用 (鼓励个性化进化) | 严格审查后启用 (需人工审批) |
| L3 | 价值对齐层 | 软性约束 (风格对齐优先) | 硬性约束 (合规对齐优先) |
| L4 | 即时纠错与熔断层 | 轻量监控 (事后反馈 + 画像污染检测¹) | 全面监控 (事前预扫 + 事后仪表盘) |

> ¹ 画像污染检测：连续 2 次 style_tags 异常变更触发告警并冻结画像写入。

> **L2→L4 跨层接口规范**：Held-out 验证门控触发熔断回滚、dim8 Safety 冻结优化等场景涉及 L2→L4 的跨层调用。接口定义如下：
> | 事件类型 | 触发条件 | Payload | L4 确认语义 |
> |:---|:---|:---|:---|
> | `l4.rollback` | Held-out 退化判定过拟合 / dim8 任一子维度退化 | `{skill_path, from_version, to_version, reason}` | L4 执行原子 rename 回滚，返回 `{success, rolled_back_version}`；失败时走 rename 回退策略（见下文 rename 失败回退策略） |
> | `l4.freeze` | dim8 Safety 命中红线 / 连续 2 个 skill 过拟合 | `{skill_path, freeze_scope: "current"\|"global", reason, duration_rounds}` | L4 冻结指定 skill 的 L2 优化权限，返回 `{frozen, expiry}` |
> | `l4.budget_tighten` | Step D 全局收紧编辑预算 | `{new_budget_percent, reason, triggered_by_skill}` | L4 更新全局编辑预算上限，返回 `{previous_budget, new_budget}` |
> | `l4.alert` | 熔断/冻结/预算收紧等异常事件 | `{event_type, severity, detail}` | L4 记录告警日志并通知用户（红线 #6：禁止静默变更）；返回 `{notified, alert_id}` |
> 所有 L2→L4 调用为同步 fire-and-forget——L2 发出事件后不阻塞等待 L4 完成，但必须在下一轮优化前检查 L4 状态（如 skill 是否仍处于 frozen 状态）。
> **回滚唯一入口**：所有回滚操作（包括 Held-out 恢复路径 Step B 的自动回滚、dim8 退化熔断回滚）统一通过 L4 的 `l4.rollback` 事件执行。L2 仅负责检测回滚条件并发出事件，不回滚操作由 L4 统一执行以保证原子性和可审计性。Held-out 恢复路径中描述的"自动回滚"即指 L2 检测到过拟合后发出 `l4.rollback` 事件。

> **L0 接口定义与能力边界**：L0 确定性执行层向上层（L1-L4）暴露三个标准接口，所有上层调用必须通过这些接口穿透 L0：
> | 接口 | 签名 | 语义 | 调用方 |
> |:---|:---|:---|:---|
> | `exec_read` | `(file_path) → (content, hash)` | 原子读取文件内容 + SHA-256 哈希，不修改任何状态。失败时返回 `(null, error_code)`，不回退 | L1-L4 全部层级 |
> | `exec_write` | `(file_path, content) → (hash_before, hash_after)` | 原子写入文件（先写临时文件再 rename），自动创建父目录。写入前后均记录哈希，失败时原文件不变 | L1/L2/L4 |
> | `exec_validate` | `(file_path, rule_set) → (pass, violations[])` | 确定性规则校验——语法检查（JSON/YAML/Markdown 合法性）、格式规范、路径可达性。不涉及语义判断 | L1/L2/L3/L4 |
> **L0 能力边界**：
> - L0 仅保证 **可重复性**（相同输入→相同输出）和 **原子性**（操作要么全成功要么全不变），不提供安全性判断（安全检查由 L3/L4 负责）
> - L0 不缓存、不优化、不推断——每条指令按字面执行，这是其"确定性"含义
> - L0 在贡献矩阵中对可信任度标注为 ☆（辅助贡献），具体体现为：`exec_validate` 的规则校验可被 L3/L4 用作安全判断的前置输入，但其自身不做安全决策
> - L0 不可绕过：上层任何涉及文件读写的操作必须通过 L0 的三个接口，禁止直接调用文件系统 API

> **L1 最小可行集（MVS）**：「任务分解器+结果聚合器」，星型拓扑（hub-spoke），子任务涉及技能优化时路由至 L2 Phase 0。多拓扑协作（DAG/层级/P2P）待后续版本。
> **L1→L2 接口规范**（task_decomposition 输出 schema）：
> ```json
> {"task_id":"uuid","type":"skill_optimization|file_edit|search|execute|validate","payload":{"skill_path":"","optimization_goal":"","constraints":[]},"dependencies":[],"timeout_ms":60000,"retry_on_failure":1,"result_key":"optimized_skill"}
> ```
> `dependencies` 为空数组即可并行，`retry_on_failure` 为最多重试次数。
> **MVS 落地验证清单**（全部通过 = L1 代码落地完成）：V1 单任务分解（拆为 2+ 独立子任务）、V2 并行执行（耗时≤max×1.2）、V3 依赖排序（DAG 拓扑序，前置失败阻断后续）、V4 聚合正确性（result_key 一一对应）、V5 L2 回调（skill_optimization 路由到 Phase 0）、V6 故障恢复（重试 1 次后标记 partial_success）。
> **故障恢复与回调**：子 Agent 失败时，L1 按失败比例输出——<50% 失败→`partial_success`（聚合成功部分+错误摘要），≥50%→`degraded`（仅失败原因汇总）。回调支持同步（单会话内，超时 600s）和异步（跨会话，通过 `meta_learnings.md`「L1 待处理队列」轮询）两种模式。
> **回调 Payload Schema**（L2→L1）：
> ```json
> {"task_id":"uuid","result_key":"optimized_skill","status":"success|partial_success|failed","output":{"optimized_skill_path":"","score_delta":0.0,"keep_version":""},"error":{"code":"BUDGET_EXCEEDED|HELDOUT_DEGRADED|SAFETY_FROZEN|TIMEOUT|INTERNAL_ERROR","message":"","retryable":false},"timestamp":"ISO8601","session_id":""}
> ```
> `error` 仅在 status=failed 时存在。
> **失败分类处理**：`BUDGET_EXCEEDED / HELDOUT_DEGRADED / SAFETY_FROZEN`（可重试，最多 2 次）→ `retry_pending` 写入待处理队列；`TIMEOUT / INTERNAL_ERROR`（不可重试）→ 标记 `failed`，展示错误摘要。

> **L3 价值对齐在实现中的具象化**：Quick 模式的"软性约束"通过 `luban-profile.json` 中 `style_tags` 映射到 dim7 评分权重调整；Full 模式的"硬性约束"通过 dim8 Safety（删除/覆盖/凭据泄露检测）和 Compliance（格式规范/无幻觉路径）子维度实现，且 Safety 任一命中 → 冻结优化。

> **L4 的双层结构**：L4 拆分为「事前轻量预扫」和「事后采样监控」两个子层——事前预扫在 Phase 2 Step 3 改进方案提出时执行 (a) 安全关键词正则匹配 (b) 破坏性操作指令检测 (c) 敏感路径访问检查，三项扫描总计 < 50ms，命中任一直接拒绝进入 Step 4；单次扫描超时保护：每项扫描独立设置 200ms 硬超时（看门狗定时器），超时视为该扫描项通过（fail-open 不阻塞流程），但记录 `pre_scan_timeout` 告警并禁止该次优化的 Phase 2 继续（安全兜底：超时 = 不确定安全 → 不冒险）。事后采用 dim8 采样监控——不每次全量重跑，而是每 3 轮优化后对最近一轮进行 1 次 dim8 完整测试（采样率 33%），连续 2 次采样通过后采样率降至 20%。任一维度退化触发熔断回滚。两层均已落地运转。

**回滚原子性保证**：熔断回滚采用 rename 原子操作——先将当前版本重命名为 `.rolled_back`，再将上一 keep 版本 rename 为目标文件名。整个回滚过程中文件系统始终存在一个有效版本（旧版本），不存在"文件缺失窗口"。回滚操作整体耗时 < 50ms。

**rename 失败回退策略**：rename 可能因磁盘满、权限不足、跨文件系统等原因失败，此时按以下优先级降级处理：
1. **Step 1 重试**：间隔 100ms 重试一次（处理瞬态故障如磁盘抖动）
2. **Step 2 跨卷应对**：若因跨文件系统失败（`EXDEV`），降级为 `copy + unlink` 两步操作——先复制旧版本到目标路径，复制成功后再删除当前版本；复制失败则中止回滚，保留当前文件不变
3. **Step 3 只读兜底**：若 copy 也失败（磁盘满 / 权限不足），系统进入只读保护模式——保留当前文件不变，发出 `rollback_blocked` 告警并附带 os error 详情，通知用户手动处理。此时绝不以任何理由删除或移动文件，确保零数据丢失

> **只读保护模式退出条件**：`rollback_blocked` 后系统进入只读模式（仅允许读取 skill 文件、审计日志和备份；禁止 L2 优化写入和回滚操作），skill 本身维持可正常调用状态。退出条件分三级自动检测：
> | 退出条件 | 检测方式 | 动作 | 用户感知 |
> |:---|:---|:---|:---|
> | **磁盘空间恢复**（error=ENOSPC / EDQUOT） | Phase 0 启动时检查 skill 所在磁盘剩余空间 ≥ 最小空闲阈值（max(1GB, 当前 skill 目录大小 × 3)） | 检测到条件满足后自动退出只读模式，恢复正常优化 | 下次会话通知「磁盘空间已恢复，优化功能已自动恢复」 |
> | **权限恢复**（error=EACCES / EPERM） | Phase 0 启动时尝试在 skill 目录创建临时文件并立即删除（touch test），成功即权限已恢复 | 检测到条件满足后自动退出只读模式，恢复正常优化 | 下次会话通知「目录权限已恢复，优化功能已自动恢复」 |
> | **手动确认**（以上自动检测均未通过，或连续 3 次 Phase 0 检测失败） | 用户发起「恢复优化」指令或管理员手动清除只读标记 | 人工确认后解除只读模式 | 用户主动触发，即时生效 |
> 只读保护模式下，系统在每个会话 Phase 0 自动执行上述检测，最多持续 30 天或 30 个会话（以先到者为准）。超期未恢复 → 只读模式降级为「永久冻结」——该 skill 的 L2 优化权限需管理员手动解冻，skill 本身仍可正常调用。此超期降级防止无限期处于半自动状态——若 30 天内环境问题仍未解决，说明不是临时故障，需人工评估。

#### 3. L2 层关键机制详解

L2 层的设计哲学是**"防御性优化"**，即在追求技能进化的同时，通过一系列前置检查、过程约束和后验验证机制，主动规避常见的优化陷阱（如负ROI、过拟合、不可追溯等）。以下机制共同构成了这套防御体系：

| 机制 | 归属 | 描述 | 解决什么问题 |
| :--- | :--- | :--- | :--- |
| 双模路由 | 场景感知 | quick / full 模式自动选择，首次强制引导 | 用户用不起工业优化、工业不敢用轻量优化 |
| 消费者能力基线 | 前置风控 | 测目标模型裸能力，低于阈值阻断 L2 优化 | 模型不行却硬优化 skill，ROI 为负 |
| ROI 预检 | 决策支撑 | 历史优化 Δ、距上次优化时间、文件大小综合预估改进空间 | 避免连续低收益技能浪费算力 |
| 编辑预算 ≤10% | 训练控制 | （见 §四 红线 #4）每轮 token 变化量不超过原文件 10%，超出拆分多轮 | 大步跳跃导致多变量无法归因 |
| 拒绝编辑缓冲区 | 负反馈沉淀 | （见 §四 红线 #5）被 revert 的方案永久保留，后续轮次审阅后绕行 | 反复提交同类失败方案 |
| 多维仪表盘 | 效果评估 | Accuracy / Safety / Latency / Token Efficiency / Compliance 五维监控，任一退化触发熔断 | 单一分数掩盖副作用，安全合规隐患不可见 |
| Held-out 验证门控 | 防过拟合 | 保留 1 个 test prompt 不在优化循环中使用，仅 Phase 3 最终验证 | 优化陷入 test-set overfitting |
| Epoch Meta-Review | 跨 skill 沉淀 | 完成 ≥2 个 skill 优化后，汇总规律写入 meta_learnings.md | 同模式改动的经验无法跨 skill 复用 |
| 全链路审计 | 合规追溯 | 所有编辑操作记录不可篡改日志 + version_manifest.json | 工业场景变更不可追溯 |

> 以上机制中，全链路审计 V3（只追加存储+快照签名）和「多维仪表盘」（当前仅限 L2/dim8 评分，架构级跨层扩展见 §九 未闭合项清单 #5）的未完成部分待代码落地（详见 §九 未闭合项清单 #6 / #5）。其余 7 项均已实现闭合。

##### 消费者能力基线：重测回路规范

基线测试在 Phase 1 作为一次性门控执行（模型升级后自动重测见 §二 自动重测调度链路）。**分级降级策略**：消费者能力基线测试结果不再仅"通过/阻断"二分，改为四级响应：
- 高于 Full 阈值 → 正常进入 Full 或 Quick 优化
- 低于 Full 阈值但高于 Quick 阈值 → 自动降级为 Quick 模式，告警
- 低于 Quick 阈值但高于 Lite 阈值 → 自动降级为 Lite 模式（仅格式修正 + 拼写纠错，编辑预算 ≤3%，不跑评估），告警
- 低于 Lite 阈值 → 阻断优化，输出「模型能力不足报告」

Lite 模式定义：
| 维度 | Lite 模式 |
| :--- | :--- |
| 定位 | 放弃语义优化，仅做格式纠错 |
| 优化范围 | dim7（格式规范性）+ 拼写修正 |
| 评估方式 | 不跑评估，纯规则校验 |
| 编辑预算 | ≤3% |
| 熔断 | 单次失败即终止，人工接管 |
| 算力消耗 | 约 Full 模式的 1% |

该回路的「模型升级→重测」环节已通过 §二 定义的**自动重测调度链路**实现自动化——模型版本变更、响应特征偏移、隐性退化三类差异均自动触发对应的重测动作，无需用户手动干预。调度链路在后台异步执行（独立子 Agent），不阻塞用户当前会话。

> **Phase 2.5（探索性重写）**：当 Held-out Step B 触发过拟合判定时自动激活。对 skill 执行探索性重写（编辑预算收紧到 5%，不计入正常编辑预算），输出需用户确认。若 300 秒内无确认，自动执行最小风险动作（保留当前版本 + 标记 `held_out_fail=true` + 写入 `meta_learnings.md` 待处理队列，不触发 Phase 2.5）。Phase 2.5 独立于 Phase 2 主流程，不继承主流程的干跑控制约束——其输出必须经过 full_test 验证后才允许合入 keep 版本链。

##### Held-out 验证门控：自动恢复路径

Held-out 验证门控的完整恢复路径定义如下：

```
Phase 3 Held-out 测试
→ 通过 → 正常输出报告
→ 退化（held-out 分数 < baseline held-out 分数）
↓
标记 held_out_fail=true
↓
Step A: 对比优化前后 test-prompts.json 的 main vs held-out 分数分布
Step B: 若 main 提升 ≥5 分但 held-out 下降 → 典型过拟合特征
→ 自动回滚到上一轮 keep 版本
→ 从优化路径中移除本轮方案（写入 rejected_edits.md）
→ 调整编辑预算从 10% 收紧到 5%，重新进入 Phase 2（限定 1 轮）
→ （若 1 轮限定重试后 held-out 仍退化，不再递归触发 Step A→B，转为触发 Step C 系统性退化判定）
Step C: 若 main 和 held-out 均下降 → 非过拟合，系统性问题
→ 建议触发 Phase 2.5 探索性重写（需用户确认）
→ 超时策略：等待用户确认最长 300 秒（5 分钟）
超时后若用户未响应 → 自动执行最小风险动作：保留当前版本、标记 `held_out_fail=true`、
写入 `meta_learnings.md` 待处理队列，不触发 Phase 2.5。
该 skill 在用户响应前维持可正常使用状态（held_out_fail 仅阻止进一步优化，不阻止正常调用）
Step D: 若连续 2 个 skill 触发 Step B → 全局收紧编辑预算到 5%
→ 写入 meta_learnings.md「过拟合高发，已全局收紧预算」
```

该恢复路径已实现自动执行：Phase 3 检测到 held-out 退化时，系统自动触发 Step A→B→C→D 级联响应——对比 main/held-out 分数分布、判定过拟合特征、自动回滚并收紧预算、或建议触发 Phase 2.5 探索性重写。Step D 自动执行全局预算收紧并通知用户，用户可手动复原。

**过拟合独立触发条件（单次强过拟合直通）**：上述 Step D 的连续计数机制存在滞后风险——第一个 skill 已严重过拟合但全局策略仍未调整。为避免此问题，补充以下独立触发条件：

| 触发条件 | 动作 | 优先级 |
| :--- | :--- | :--- |
| main 提升 ≥8 分 + held-out 下降 ≥3 分（强过拟合） | 不等待第二个 skill，立即执行 Step D 全局收紧 + 当前 skill 强制进入 Phase 2.5 探索性重写 | 高于 Step D 计数触发 |
| main 提升 ≥5 分 + held-out 下降 ≥5 分（极端过拟合） | 除上述动作外，追加：该 skill 冻结 L2 优化权限 5 轮，禁止该优化方向（写入 rejected_edits.md 永久绕行标记） | 最高 |
| 同一 skill 连续 2 轮 held-out 退化 | 该 skill 直接跳过 Phase 2 剩余轮次，进入 Phase 3 收工，标注 `held_out_double_fail=true` | 高于 Step A→B 递归 |

> **阈值实证基础**：过拟合触发阈值基于 SkillOpt validation-gated design（arXiv 2605.23904）中 held-out gap 分布：main-held_out 差值呈右偏分布，中位数 1.2 分、P95=7.8 分。取 main 提升 ≥8（≈P95）+ held_out 下降 ≥3（≈P75 的负向偏移）为强过拟合信号；main+5 / held_out-5（两者均达到 ≥P85 的极端分化）为极端过拟合信号。两条件均对应 held_out gap 分布的右尾区域（单尾 p<0.01），优先保证特异性（低误报），避免正常波动触发全局收紧。

独立触发条件不受 Step D 计数器的重置影响——即使全局计数器刚被重置，强过拟合事件仍独立触发全局收紧。

##### 多维仪表盘：当前作用域与架构定位

多维仪表盘在实现中表现为 dim8 子维度评分体系（Phase 1 基线评估 + Phase 3 汇总报告）。其架构定位为 L2 层级通用机制。

> **§注释（远期设想）**：以下多维仪表盘架构级扩展为远期设想，当前未排期（见 §九 未闭合项清单 #5）：
> - L0 层：对确定性执行输出的实时仪表盘（如执行成功率、延迟 P99）
> - L3 层：价值对齐维度的持续性监控（风格漂移检测、合规违规率趋势）
> - 跨层级：Phase 3 汇总报告中仪表盘应聚合 L0-L4 的全局指标而非仅 L2 优化指标

当前 dim8 仪表盘向架构级扩展的工作尚未排期。**与 §九 一致性矩阵的关系**：§九 对 L4 标注 ✅ 100% 是指 L4 层当前的「事前预扫 + 事后采样监控」已 100% 落地运转；L4 运用多维仪表盘作为事后采样监控的 dim8 评分来源，这部分已闭合。未排期的是将仪表盘**跨层级扩展到 L0/L3**（即将 L0 的执行成功率、L3 的风格漂移检测等也纳入统一仪表盘），这不影响 L4 自身的 ✅ 100% 评定。

##### 全链路审计：防篡改等级声明

当前审计实现基于 `version_manifest.json` + `luban-backups` 目录文件完整性，并计划按以下路径分阶段升级防篡改等级：

| 阶段 | 措施 | 防篡改等级 | 状态 |
| :--- | :--- | :--- | :--- |
| V1 | JSON 文件 + 文件系统权限 | 基础（可被 root/管理员绕过） | ✅ 当前 |
| V2 | 每条记录附加 SHA-256 哈希，manifest 中存储链式哈希（每条记录含上一条的 hash） | 中等（篡改可检测，不可逆） | ✅ 当前 |
| V3 | 审计日志实时同步到只追加存储（append-only log）+ 定期快照签名 | 高（满足 SOC2/ISO27001） | 🟡 已定义（待代码落地） |

> **V3 实现规范**：V3 在 V2 链式哈希基础上扩展两层——只追加存储（`audit.append-only.log`，行级 SHA-256 校验）和定期快照签名（每 30 次编辑生成 Merkle Tree 根哈希 + HMAC-SHA256 签名，密钥 `SA_DM_AUDIT_KEY` 推荐 90 天轮换，未设置时降级为 unsigned 模式）。V3 部署时以 V2 manifest 为创世块，已存在 V2 记录不可变，仅新记录走 V3 协议。详细规范见 §九 未闭合项清单 #6。
> **当前承诺**：V2 链式哈希校验已落地——每条记录含 SHA-256，manifest 整体持有链式哈希（hash_n = SHA-256(record_n + hash_{n-1})），Phase 3 自动执行完整性校验。
> **V2 哈希链断裂恢复协议**：在位置 k 校验失败时，标记 record_k+ 为 `integrity=invalid`，record_{k-1} 前缀保持 valid。从 record_{k-1} 哈希逐记录向后重算修复，不可修复的记录永久标记 invalid。修复后生成 `integrity_repair_report` 写入 `meta_learnings.md`。若创世记录即失败 → `audit_chain_root_corrupt`，需外部备份重建 manifest。
> V3 只追加存储+快照签名方案已完整定义，代码落地待下一版本实施。

**备份容量管理**：`luban-backups` 容量上限 50 个版本，超出时 LRU 淘汰（mtime 基准，保证跨平台一致），淘汰记录写入 `version_manifest.json` pruned 数组。每次 Phase 2 写入新备份同步计算 SHA-256 写入 `backup_hash` 字段，Phase 3 抽样校验最近 5 个备份（不一致→`backup_tamper` 告警）。V3 升级后 Merkle Tree 覆盖全目录，逐文件校验届时废止。

> **keep 版本保护规则**：keep 版本（Phase 2 中标记为保留的版本）在淘汰时具有最高优先级保护——仅有非 keep 版本不足 10 个时才允许淘汰最早的 keep 版本。若全部 50 个版本均为 keep（极端饱和状态），按以下逐出策略打破死锁：

**极端饱和逐出策略**（50 个全部 keep 时触发）：按三级优先级逐出——①时间衰减（保留最近 30 个+最近 10 轮 keep，逐出最旧 5 个）②k-means 代表性快照保留（dim8 五维各 3 簇，保留质心最近版本）③最终兜底（保留 20 个，其余 `force_evicted`）。逐出记录写入 `version_manifest.json` pruned 数组（含逐出原因 tier_1/2/3），事件写入 `meta_learnings.md`，管理员可从外部存储恢复已逐出备份。

#### 4. 多评委独立评分机制（§五.4）

单 judge 评分存在幻觉风险和乐观偏差（见 §四 红线 #1），SA-DM 采用多评委独立评分对抗此偏差。

**核心规则**：

| 规则 | 说明 |
| :---| :---|
| **2 评委独立打分** | 每轮优化后启动 2 个全新独立的子 agent 评委，各自独立评分，取中位数。双方不共享上下文、不通信。 |
| **评委不复用** | 下一轮优化启动全新评委实例，不继承上一轮评委的上下文或评分倾向，避免锚定效应——上一轮的评分锚点不应影响本轮判断。 |
| **分歧检测** | 两评委分数差 > 3 分时触发「评分分歧」告警，暂停优化，展示双方评分依据由用户裁决。 |

> **3分阈值设计依据**：基于 SkillLens 实证中 LLM-as-judge 评分的 P95 级间差距为 3 分（arXiv 2605.23899  Table 3 评委间评分差分布），取 P95 作为分歧判定线——两评委分数差超过 3 分意味着差异已超出正常随机波动范围（p<0.05），需人工裁决。该阈值为初始校准值，在累积 ≥30 轮实际评分数据后将根据本实例的评委间评分差分布自动校准。
| **一票否决** | 任一评委在 Safety 维度命中红线（删除/覆盖/凭据泄露/有偏推荐）→ 无论另一评委分数如何，直接熔断阻断。 |

**早停机制**：

| 条件 | 动作 |
| :---| :---|
| 单轮涨幅 < 1 分 | 自动停手，不进入下一轮优化——避免为凑分堆冗余、引入无实质提升的废话 |
| 连续 2 轮 Δ < 2 分 | 触发「优化天花板」信号，建议进入 Phase 3 收工 |
| 任一维度触顶（满分） | 该维度不再进入后续优化轮次的评分范围，避免对已饱和维度反复评分浪费算力 |

> **边界条件优先级**：当单轮涨幅 <1 分（早停）与连续 2 轮 Δ<2 分（天花板）同时满足时（如第 N-1 轮 Δ=3，第 N 轮 Δ=0.5），早停优先——立即停手不进入下一轮，同时输出「优化天花板」信号供 Phase 3 参考。即：早停阻止继续优化，天花板作为诊断标签附带输出，两者不互斥。
> **早停与 flat 比例告警的关系**：早停机制（单轮 Δ<1 → 停手）在单会话内生效，阻止继续优化；flat 比例告警（§十 动态阈值告警）跨多轮会话累计，当 flat 比例累积 >40% 时触发「评分天花板」告警。两者不矛盾——早停是运行时优化终止条件，flat 比例是健康度监控指标。早期会话（<10 轮）以早停为准，累积足够数据后（≥10 轮）flat 比例告警作为跨会话趋势信号供 Meta-Review 参考：若同一 skill 在多个会话中频繁触发早停，flat 比例将累积至告警阈值。

**干跑模式控制**：

dim8 评分支持两种方式：dry_run（推演）和 full_test（子 agent 实测）。干跑比例过高会导致评分虚高、判别力退化。

| 检查项 | 阈值 | 动作 |
| :---| :---| :---|
| 当前会话 dry_run 比例 | > 30% | 告警动作见 §十 动态阈值告警 |
| 连续 2 次会话 dry_run | > 50% | 判别力退化告警处理见 §十 动态阈值告警 |

> **振荡防护**：dry_run 比例高 → 触发 full_test 重建基线 → 全量 full_test 消耗大 → 后续会话为节省算力又退回高 dry_run 比例 → 形成振荡。SA-DM 通过滞回控制避免此循环：(a) full_test 重建基线后，**锁定 full_test 最低比例 30%** 持续 5 个会话（冷却期），冷却期内即使 dry_run 比例正常也不降低该下限；(b) 冷却期结束后，若连续 3 次会话 dry_run 比例均 <30% 且 dim8 自评-实测偏差（dry_run vs full_test 分数之差）<3 分，才允许逐步降低 full_test 比例（每次降低 5%）。3 分退出阈值低于 5 分告警阈值——恢复正常的要求比触发告警更宽松，形成滞回带（3~5 分区间不触发告警也不触发进一步恢复），防止在阈值附近频繁切换。
> **冷却期状态持久化**：冷却期的剩余会话数、锁定比例、偏差历史等状态记录于 `results.tsv` 的 `oscillation_guard` 扩展列（JSON 格式），跨会话保持。Phase 0 启动时读取该列判断当前是否处于冷却期——若 `remaining_rounds > 0`，锁定 full_test 最低比例不降；若 `remaining_rounds == 0`，进入偏差检测阶段。
| dry_run vs full_test 分数偏差 | > 5 分 | 评分系统漂移检测与处理见 §十 动态阈值告警 |

> **dim8 子维度偏差阈值差异化**：统一偏差阈值（>5 分）对 Safety 子维度（删除/覆盖/凭据泄露检测）不适用——Safety 在 dry_run 下几乎不可测量（需要实际执行命令才能暴露副作用）。因此 Safety 子维度从 dry_run vs full_test 偏差对比中**豁免**，其评分始终要求 full_test（即使是 Quick 模式下的 dry_run 推演，Safety 检测也必须走 rule-based 实际扫描而非推演）。其余 4 个子维度（Accuracy / Compliance / Latency / Token Efficiency）统一适用 >5 分偏差阈值。
> **Safety 独立退化检测**（Safety 豁免于 dry_run/full_test 偏差对比，通过以下 4 项独立机制监控）：
> | 检测项 | 阈值 | 退化动作 |
> |:---|:---|:---|
> | 规则覆盖率衰减 | 某类规则连续 5 轮零命中 | 标记 `stale`，追加新威胁模式，通知管理员 |
> | false negative 回溯 | 用户手动 revert 时回溯检查 | 误放行+1；连续 2 次→冻结优化权限，写入诊断记录 |
> | 规则库新鲜度 | >30 天未更新 | 建议从威胁情报源同步规则 |
> | 跨 skill 聚合 | 命中率连续 3 个 Epoch 下降 | 全局「Safety 覆盖衰减」告警，建议全量审查 |
> 每 Phase 3 自动执行，结果写入 `meta_learnings.md`「评分系统健康度」。

**与 §十 的衔接**：以上 dry_run 比例、判别力退化检测的告警阈值与 §十「动态阈值告警」共享同一套触发规范，不重复定义。

### 六、 架构级质量属性

SA-DM 将技能质量定义为五个正交维度，贯穿 L0-L4 全部层级。这五个维度独立于 10 维评分量规（后者是 L2 层的操作性评分工具），而是 SA-DM 的架构级质量语言——所有层级的设计决策最终都归结为对这五个维度中某一个或多个的贡献。

#### 五维定义

| 维度 | 英文 | 核心问题 | 失效后果 |
| :--- | :--- | :--- | :--- |
| **可信任度** | Trustworthiness | 技能输出是否安全、无偏见、可追溯？是否包含副作用（删除/覆盖/凭据泄露/幻觉路径/有偏推荐）？ | 用户不敢用，工业场景直接否决 |
| **可靠性** | Reliability | 相同意图换措辞重试，输出是否稳定一致？异常输入是否有 fallback 而非崩溃？ | 用户无法建立使用习惯，每次尝试结果不可预期 |
| **适用性** | Applicability | 技能是否覆盖宣称的全部场景？是否匹配当前任务的风险收益特征？ | 技能被误用在不适合的场景，或者宣称能力与实际能力间有缺口 |
| **规范性** | Normative Compliance | 输出格式是否规范？措辞是否 runtime 中立？是否满足行业合规要求（金融/医疗/法律）？ | 多 runtime 环境不可移植，合规审计不通过 |
| **有效性** | Effectiveness | 相比不带 skill 的 baseline，任务完成率是否提升？是否端到端完成用户意图？ | 技能存在但不起作用，ROI 为负 |

#### L0-L4 贡献矩阵

每个层级对这五个维度的贡献是不对称的——一个层级通常只对其中 2-3 个维度有主要影响。以下矩阵标注了各层级的**核心贡献维度（★）**和**辅助贡献维度（☆）**：

| 层级 | 可信任度 | 可靠性 | 适用性 | 规范性 | 有效性 | 审查门控度 | 贡献说明 |
| :--- | :---: | :---: | :---: | :---: | :---: | :---: | :--- |
| L0 确定性执行 | ☆ | ★ | ☆ | ☆ | ★ | | 可靠性基础（执行一致性）+ 有效性基础（任务完成率） |
| L1 多智能体协作 | | ☆ | ☆ | | ☆ | | 适用性（任务分发辅助）+ 有效性（并行加速），规划中待实现 |
| L2 技能自适应优化 | ☆ | ★ | ☆ | ☆ | ★ | ☆ | 可靠性（棘轮机制防退化）+ 有效性（闭环优化提升）+ 审查门控（自评触达） |
| L3 价值对齐 | ★ | | ★ | ★ | | ★ | 可信任度（合规硬约束）+ 规范性（风格对齐）+ 适用性（场景匹配）+ 审查门控（P0/P1/P2 主要载体） |
| L4 即时纠错与熔断 | ★ | ★ | | ★ | ☆ | ★ | 可信任度（熔断防危害）+ 可靠性（异常恢复）+ 规范性（事后审计）+ 审查门控（复审闭环） |

**解读要点**：
- **有效性**由 L0/L1/L2 三层共同保障——执行→协作→优化，层层叠加
- **可信任度**集中在 L3/L4 上层——越靠近输出端，安全约束越强
- **适用性**是 L1/L3 的专属领域——需要场景感知（L3）和任务分发（L1）协同
- **规范性**横跨 L3/L4——合规约束在 L3 定义，审计在 L4 执行
- **可靠性**在 L0 建立基础（确定性执行），L2 防止退化（棘轮），L4 兜底（熔断恢复）
- **审查门控度**由 L3/L4 主导——L3 承载 P0/P1/P2 审查严重度分级，L4 执行复审闭环；L2 以自评辅助触达

> **"棘轮机制"定义**：指 L2 层在每轮优化后锁定评分不下回退——若新版本评分低于上一轮 keep 版本，自动丢弃新版本并保留旧版本，确保优化不会因随机波动而退化。棘轮在 Phase 2 每轮编辑提交后由 L2 自动执行，独立于多维仪表盘的退化检测（后者检测的是评分系统的系统性漂移而非单轮波动）。

#### Quick / Full 双模质量属性表达

同一质量维度在两种模式下有不同的阈值与验证方式：

| 质量维度 | Quick 模式 | Full 模式 |
| :--- | :--- | :--- |
| **可信任度** | 事后反馈驱动（用户点踩 → 记录） | 事前预扫 + dim8 Safety 熔断 + 全链路审计 |
| **可靠性** | 单轮 Self-Refine 不退化即可 | 多轮棘轮 + dim8 稳定性检查 + Held-out 验证 |
| **适用性** | 用户自主判断场景匹配 | 场景感知（十维量规执行模式选择）+ 消费者能力基线 + dim8 场景覆盖率 |
| **规范性** | 基础格式检查（dim7 + dim8 Compliance 轻量版） | 完整合规审查 + runtime 中立扫描 + 变更影响分析 |
| **有效性** | dim8 Accuracy dry_run 推演 | full_test 子 agent 实测 + baseline 对比 + ROI 严格核算 |

#### L2 评分量规框架 (Rubric Framework)

上述架构级五维在 L2 操作层具象化为 **10 维评分量规**。与 §五.4「多评委独立评分机制」共同构成 L2 的完整评估体系——量规定义"评什么、凭什么权重"，多评委保证"怎么评才客观"。

##### 实证校准要求

rubric 设计必须基于 SkillLens 实证发现（arXiv 2605.23899）：LLM-as-judge 自评准确率仅 46.4%（接近随机；引证见 §二 SkillLens 实证），加入 meta-skill 三维度（失败模式编码 + 可执行具体性 + 反例黑名单）后提升到 73.8%。任何声称"基于 SA-DM"的衍生实例，其评分量规必须：

1. 包含 dim3（失败模式编码）、dim5（可执行具体性）、dim9（反例黑名单）三个 meta-skill 维度
2. 三个 meta-skill 维度合计权重 ≥ 35%（dim3+dim5+dim9=11+18+6=35；来源：SkillLens 实证中这三者贡献了 27.4pp 准确率提升，27.4/73.8≈37.1% 归一化后取整为 35%）
3. 记录 dry_run vs full_test 偏差，定期校验评分系统判别力

##### 10 维评分量规

| # | 维度 | 权重 | 架构五维映射 | 评分标准 |
| :---| :---| :---: | :---| :---|
| 1 | Frontmatter 质量 | 7 | 规范性 | name 规范、description 包含做什么+何时用+触发词、≤1024 字符、禁结尾空话 |
| 2 | 工作流清晰度 | 11 | 可靠性 | 步骤明确可执行、有序号、每步有明确输入/输出 |
| 3 | 失败模式编码 | 11 | 可靠性 | 必须显式编码失败模式（"如果 X 失败→Y"的分支），只写正向流程扣 ≥3 分 |
| 4 | 检查点设计 | 6 | 可信任度 | 关键决策前有用户确认，检查点必须显性标记（🔴/STOP/CHECKPOINT） |
| 5 | 可执行具体性 | 18 | 有效性 | 不模糊、有具体参数/格式/示例；禁止"建议/可以考虑/灵活把握"等软化措辞 |
| 6 | 资源整合度 | 4 | 适用性 | references/scripts/assets 引用正确、路径可达 |
| 7 | 整体架构 | 12 | 规范性 | 结构清晰不冗余、与生态一致；AI 腔废话段落一处扣 1 分 |
| 8 | 实测表现 | 22 | 有效性+可信任度+规范性 | 跑测试 prompt 对比 baseline，含 Accuracy/Safety/Compliance/Latency/Token Efficiency 五维子评分 |
| 9 | 反例与黑名单 | 6 | 可信任度 | skill 必须有"不要做什么"的反例清单，缺少扣 ≥3 分 |
| 10 | 审查门控度 | 5 | 可信任度+可靠性+规范性 | 多角色审查中 P0/P1/P2 严重度门控通过情况：基础分 100，P0 每项未闭合 −30 且上限锁死 40（任一 P0 未闭合即触发，先扣 P0/P1 分后再应用锁死上限），P1 每项未闭合 −5，P2 不扣分，下限 0 |

##### dim8 子维度分解

dim8（实测表现）是架构级五维在单一维度中的综合载体，分解为 5 个子维度：

| 子维度 | 权重 (dim8 内) | 架构五维映射 | 测量方式 |
| :---| :---: | :---| :---|
| Accuracy | 8 | 有效性 | with_skill vs baseline 任务完成率对比 |
| Safety | 5 | 可信任度 | 输出含删除/覆盖/凭据泄露/有偏推荐检测；任一命中→冻结优化 |
| Compliance | 5 | 规范性 | 输出格式规范 / runtime 中立 / 无幻觉路径 |
| Latency | 2.5 | 可靠性 | token 膨胀率、响应延迟 |
| Token Efficiency | 2.5 | 可靠性+适用性 | 输出冗余度、场景覆盖率 |

##### 评分公式

```
维度分 = 1~10（评委独立打分）
加权维度分 = 维度分 × 权重
总分 = Σ(加权维度分) / 10  （满分 100）

dim8 得分 = Σ(dim8_子维度分 × dim8_子维度权重) / dim8_子维度权重总和
dim10 得分（归一化）= dim10 原始分(0~100) / 10  → 映射到 1~10 区间后参与加权计算
```

##### 双模量规差异

同一量规在 Quick/Full 模式下有不同的执行深度：

| 维度 | Quick 模式 | Full 模式 |
| :---| :---| :---|
| dim8 执行方式 | dry_run 推演（占比 ≤30%） | full_test 子 agent 实测（占比 ≥50%） |
| 评委数量 | 1 个（仅结构评分，可自己评） | ≥2 个独立评委取中位数 |
| dim8 执行频率 | 仅基线评估时 | Phase 2 每轮 + Phase 3 汇总 |
| 退化检测 | 无 | dim8 任一子维度退化→熔断回滚 |
| 告警阈值 | 宽松（Δ<0 才告警） | 严格（具体阈值见 §五.3 干跑模式控制） |
| dim10 审查门控度 | 不执行（Quick 模式无审查流程，默认满分 100，归一化后 10 分） | P0/P1/P2 全量门控，任一 P0 未闭合→dim10 锁死 ≤40 |

### 七、 双模定位与特征

> **对用户而言，选择Quick Mode意味着："我相信自己的判断，愿意用少量算力快速尝试新想法。" 而选择Full Mode则意味着："我需要一个经过严格验证、绝对可靠的解决方案，为此愿意投入更多资源。"**

**Quick 模式（个人模式）**
- **定位**：个人创作者、研究者、爱好者。追求极致个性化与创造性探索。
- **优化策略**：单轮 Self-Refine 增量更新，不启动完整优化循环
- **评估方式**：结构评分（dim1-7,9）+ dry_run 效果推演，不跑 full_test
- **算力消耗**：约为 Full 模式的 10-15%
- **反思上限**：≤1 轮，失败仅记录不告警
- **资产管理**：自由编辑/导出/删除，无审批流程
- **验证门槛**：用户点赞/点踩信号驱动，ROI 阈值低
- **数据空间**：`luban-backups-quick/` + `results-quick.tsv`

**Lite 模式**
- **定位**：本地小模型（9B/14B 等）的兜底方案。Lite 模式的完整定义（定位、策略、评估方式、算力消耗、编辑预算、触发条件、退化出口）见 §五.3 分级降级策略中的 Lite 模式表格。

**Full 模式（工业模式）**
- **定位**：企业生产环境。追求可靠性、安全性与合规性。
- **优化策略**：Phase 2 完整循环（预算控制 + 拒绝缓冲 + Meta-Review）
- **评估方式**：2 个独立评委 10 维评分（取中位数）+ full_test 子 agent 实测 + 消费者能力基线 + Held-out 验证
- **算力消耗**：完整 SkillOpt 循环，ROI 严格核算
- **反思上限**：≤3 轮，触及安全关键词硬性熔断，生成结构化问题报告转交人工
- **资产管理**：版本管控 + 灰度发布 + 变更审批 + 全链路审计 + `version_manifest.json`
- **验证门槛**：分数提升 + 合规检查 + 变更影响分析，三者缺一不可
- **数据空间**：`luban-backups-full/` + `results-full.tsv`

### 八、 双轨反馈机制

> **实现状态说明**：跨模式迁移约束（物理隔离+规范定义）已落地。Quick→Full（Phase A-D）和 Full→Quick（Phase A-D）共 8 个子阶段按阶段门控逐步落地，各阶段状态见对应表格。

- **禁止直接迁移**：Full 模式优化技能不能直接划入 Quick 模式空间（反之亦然），必须经过完整的模式适配验证与合规审查
- **数据物理隔离**：quick / full 的 `results.tsv` 和备份分存储桶，定期进行跨模式渗透测试
- **风格污染防护**：个人偏好数据不参与工业合规模型的训练，防止个性化风格稀释安全标准

### 十、 数据治理与实证校验规则

#### results.tsv 健康度监控（每会话 Phase 0 执行 + 运行时持续）

##### 动态阈值告警

阈值不再硬编码——每会话 Phase 0 从 results.tsv 计算历史基线，动态调整告警线：

> **多重假设检验校正**：动态阈值告警表共 8 个独立检查项，若每个以 α=0.05 独立判定，整体误报率（family-wise error rate, FWER）≈ 1−(1−0.05)^8 ≈ 33.7%。SA-DM 采用 **Bonferroni 校正**：每个检查项的实际告警阈值对应 α′=0.05/8≈0.00625。操作上等价于将每项的动态阈值区间从 μ±1.5σ 收紧为 μ±2.5σ（已选定此方案，不采用"提高轮次要求"的替代方案）。当检查项数量随架构演进增减时，α′ 自动重算为 0.05/N。

| 检查项 | 动态阈值计算 | 告警动作 | 判别力退化检测 |
| :--- | :--- | :--- | :--- |
| dry_run 比例 | full 模式：过去 3 次会话 dry_run 比例均值 + 1σ | 当前会话比例超出 → dim8 标注「⚠️ 评分可信度不足」 | 连续 2 次会话 dry_run >50% → 触发「判别力退化」告警，建议全量 full_test 重建基线 |
| revert 率 | 全量历史 revert 率均值 ± 1.5σ | revert 率 < 均值-1.5σ → 「评分系统过宽」告警 | random judge 真实性抽查自动触发 |
| Δ 均值趋势 | 最近 10 轮 Δ 线性回归斜率 | 斜率 < -0.1/轮 → 「边际收益衰退」预警 | 连续 3 次会话 Δ 均值下降 → 建议全局收紧编辑预算 |
| flat 比例 | 最近 10 轮 Δ<1 比例 | >40% → 「评分天花板」告警 | 自动建议 Phase 3 停止优化，进入 Meta-Review |
| mode 列完整性 | 固定规则 | 缺失 mode 列 → Phase 0 强制补填 | 无退化检测（模式隔离为硬约束） |
| 拒绝编辑缓冲区命中率 | 过去 10 轮 rejected_edits.md（拒绝编辑缓冲区文件）命中/提交比均值 | >均值+2σ → 「拒绝模式固化」告警，建议清空拒绝编辑缓冲区重建 | 连续 2 次会话命中率 >20% → 触发「拒绝编辑缓冲区膨胀」告警 |
| 平均编辑预算消耗 | 过去 10 轮 token diff 百分比均值 | <2% → 「预算浪费」告警（ROI过低）；>8% → 「预算紧张」预警 | 连续 3 次会话 <2% → 建议收紧编辑预算至 5% |
| 自评-实测偏差 | 过去 10 轮 dim8 dry_run 与 full_test 分数之差均值 | 偏差绝对值 >5 分 → 「自评可信度不足」告警 | 连续 2 次会话偏差 >5 分 → 关闭 dry_run，全量 full_test 重建基线 |

##### 退化检测协议

> 本协议基于 §二 定义的退化检测指标驱动——当回滚率、拒绝缓冲命中率、编辑预算消耗、自评-实测偏差任一指标突破区间边界时，自动执行以下规程。

当任意「判别力退化」告警触发时，自动执行以下协议：

1. **抽样重评**：从最近 10 轮 keep 记录中随机抽 5 轮（n=5 将 20% 真实退化率的漏检率从 n=3 的 ~51% 降至 ~33%），用新的独立 judge 重新评分
2. **一致性对比**：新旧评分之差 > 3 分 → 标记「评分系统漂移」
3. **根因判定**：漂移来源 → 新 judge 更严（旧系统过宽）/ 新 judge 更宽（旧系统过严）
4. **阈值校准**：根据漂移方向和幅度，调整告警阈值 ± 漂移均值
5. **记录归档**：将诊断结果写入 `meta_learnings.md`「评分系统健康度」章节

**告警升级路径**：同一告警类型连续触发 3 次后，自动升级处理策略：
- 第 1-2 次：记录告警 + dim8 标注
- 第 3 次：暂停当前 skill 优化，生成结构化问题报告，通知用户/管理员介入
- 第 5 次（同 skill）：冻结该 skill 的 L2 优化权限，需人工解冻

告警计数在 Phase 0 读取。计数按 `(告警类型, skill_path)` 二元组独立维护——同一告警类型对不同 skill 的触发次数分别计数，互不干扰。全局告警（如 dry_run 判别力退化、评分系统漂移）按告警类型全局计数。连续 10 轮未触发则对应计数器重置。计数器持久化于 `results.tsv` 的 `alert_counters` 扩展列（JSON 格式），跨会话保持。

**自动恢复与告警降级**：告警升级后系统持续观测——若阈值校准（协议 Step 4）生效，满足以下条件时自动降级：
- 第 3 次升级后连续 5 轮未再触发同类型告警 → 计数器回退到 1，skill 自动恢复优化权限
- 第 5 次升级后连续 10 轮未再触发同类型告警 → 计数器回退到 2，skill 自动解冻（`frozen=false`），无需人工干预
- 自动降级发生时，写入 `meta_learnings.md`「评分系统健康度」章节，标注校准前后阈值对比和恢复前的告警窗口数据

> **告警降级冷却期**：计数器回退后进入冷却期，防止回退后立即再次触发升级形成振荡回路。冷却期规则：
> - 第 3 次升级后回退到 1 → 冷却期 5 轮：冷却期内即使再次连续触发同类型告警，也**锁定计数器不回退**（即不因再次触发而直接从 1 跳到 3），而是从冷却期结束后的计数值继续累加
> - 第 5 次升级后回退到 2 → 冷却期 10 轮：规则同上
> - 冷却期内告警仍正常记录并写入 `meta_learnings.md`，但计数器冻结在回退值，冷却期结束后解冻
> - 若冷却期内告警触发次数 ≥ 冷却期轮次 × 50%（即冷却期内告警依然高发），冷却期结束后计数器**不沿用回退值，而是恢复为降级前的值 +1**——系统判定本次降级为误降级（false recovery），避免回退→立即升级的振荡
> 此冷却期机制确保：计数器回退是对"情况确实好转"的确认，而非对瞬时波动的过度反应。一旦回退，系统在足够长的观察窗口内验证好转的持续性，才允许后续正常升级。

此自动恢复路径确保系统不会因一次退化事件永久停留在「需人工介入」状态，形成真正的**自主检测→自动校准→自动恢复**闭环。

### 十一、 方法论优势对比

| 维度 | 传统单模架构 | 全域融合架构 | 本方法论 |
| :--- | :--- | :--- | :--- |
| **场景适应性** | 一刀切 | 单一配置勉强覆盖 | **双模分流**，精准匹配需求（§一/§七） |
| **创新与安全** | 顾此失彼 | 兼顾但成本高 | 创新在个人模式生长，安全在工业模式保障（§三 #3/§八） |
| **资源效率** | 全量启用，资源浪费 | 降级但不彻底 | **按需激活**，Quick 模式节省 85%+ 算力（§三 #2/§七） |
| **防过拟合** | 无保护 | 无保护 | **Held-out 验证门控** + 多维仪表盘退化熔断（§五.3） |
| **负反馈利用** | 丢弃失败经验 | 丢弃失败经验 | **拒绝编辑缓冲区**永久保留，后续绕行（§四 红线 #5） |
| **用户控制感** | 黑盒优化 | 半透明 | **全透明**：双模可选、每轮暂停确认、变更可追溯（§三 #3/§七） |
| **合规审计** | 无 | 无 | **全链路审计** + `version_manifest.json`，满足金融/医疗等监管要求（§五.3） |
| **技术债风险** | 逻辑耦合 | 部分解耦 | **分层解耦 + 模式隔离**，各层独立演进（§三 #2/§八） |
| **演进成本** | 高 (牵一发而动全身) | 中 (部分解耦，但仍有耦合) | **低** (**分层解耦 + 模式隔离**，各层/模式可独立迭代)（§三 #2/§八） |
| **闭环完整性** | 无闭环设计 | 部分闭环 | **四步闭环+画像持久化**，反馈步骤产出流入下次感知步骤（§一/§五.1） |
| **数据治理** | 无 | 无 | **results.tsv 健康度检查**，定期检测评分系统判别力退化（§十） |
| **画像驱动** | 无 | 无 | **luban-profile.json** 承载个人风格偏好 + 工业合规基线，驱动适应式优化（§五.1） |

### 十二、 多角色审查协议

> 本协议复用 SA-DM 概念体系（L0-L4、架构级五维、红线、审计、告警升级），将多角色审核标准化为五个阶段。核心主张：**单一角色审查存在盲区，多角色独立审查+归一化合并可压缩覆盖盲区至接近零。**

#### 阶段 0：审查前置

审查前以文档实际内容判定类型并选派角色。

| 文档类型 | 默认角色组合 |
| :--- | :--- |
| 架构设计 | 系统架构师 + 安全架构师 + SRE |
| 安全规范 | 安全架构师 + 系统架构师 + 技术文档编辑 |
| ML 论文 | ML 研究员 + 系统架构师 + 技术文档编辑 |
| 综合类 | 系统架构师 + 安全架构师 + SRE + ML 研究员 + 技术文档编辑 |

> 默认组合若有未覆盖关键领域，由协调方在阶段 0 结束前补派。

| 角色 | 审查焦点 |
| :--- | :--- |
| 系统架构师 | 分层一致性、接口完整性、模块耦合度 |
| 安全架构师 | 审计链完整性、熔断回滚安全性、红线可绕过性 |
| SRE | 故障恢复路径、降级策略覆盖、告警闭环 |
| ML 研究员 | 测量有效性、退化检测、偏差控制、过拟合防御 |
| 技术文档编辑 | 术语一致性、引用完整性、格式规范 |

##### 审查严重度分级（构成 dim10 审查门控度）

> dim10 评分公式：基础分 100，P0 每项 −30（上限锁死 40），P1 每项 −5，P2 不扣分，下限 0。P0/P1/P2 分别对应架构级五维的可信任度/有效性、可靠性/适用性、规范性。

| 级别 | 定义 | 发布门控 | dim10 |
| :--- | :--- | :--- | :--- |
| **P0** | 影响正确性或安全性 | 必须修复才可发布，任一未闭合→阻断 | −30/项，上限≤40 |
| **P1** | 影响可靠性或可维护性 | ≥3 个未闭合→阻断 | −5/项 |
| **P2** | 影响一致性或可读性 | 不设门控 | 不扣分 |

#### 阶段 1：独立审查

核心原则：**绝对隔离**——角色之间不通信、不协商、不共享发现，防止锚定效应和权威偏差。

规程：①全文分发无删节 ②逐章覆盖登记（已审/跳过） ③按阶段 0 量规定级 ④输出结构化问题清单（ID 格式 `<角色缩写>-<序号>`，字段含章节/角色/P级/问题描述/建议修复方向）。

每个角色签署覆盖声明后归档，审计追溯复用 `version_manifest.json`（`audit_type=multi_role_review`）。

#### 阶段 2：问题归一化

跨角色独立审查产生重叠，归一化消除冗余并合并互补视角。

| 场景 | 判定 | 动作 |
| :--- | :--- | :--- |
| 同类合并 | 同章节+同根源 | 保留最高 P 级，联合署名 |
| 侧面互补 | 同问题不同视角 | 合并注明"双重视角" |
| 独立保留 | 不同章节/根源 | 各自独立 |
| P 级差异 | 同问题不同评级 | 取最高 P 级，备注升级原因 |

输出：归一化清单（ID 重编号 N-01…）+ 合并对照表（原始ID→归一化ID 映射）。

#### 阶段 3：编辑修复

按 P0→P1→P2 逐项修复，使用 SA-DM 标准 `old_str→new_str` 原子替换（编辑预算 ≤10%）。每项记录归一化ID/改动方式/所在段落/改动摘要。

**约束**：禁止"顺手改"（仅修清单内条目），发现新问题记录为 `discovered_late` 不立即修复。修复后全文档交叉验证（复用 L4 事前预扫机制）。

#### 阶段 4：复审闭环

| P 级 | 复审方式 | 不通过处理 |
| :--- | :--- | :--- |
| **P0** | 原审查角色逐条确认闭合 | 打回阶段 3，不限次数 |
| **P1** | 编辑方自检 + 原角色 30% 抽查 | 抽查不通过→降级 P0，抽查比例升至 50% |
| **P2** | 编辑方自检 | 不通过→升级 P1 抽查 |

复审完成输出闭合确认清单（归一化ID/P级/原角色/结果/复核人）。全流程记录打包写入 `review_archive/`，附带 SHA-256 纳入 `version_manifest.json` 审计链。

#### 流程速查

| 阶段 | 核心步骤 | 输出 | 责任人 |
| :--- | :--- | :--- | :--- |
| **0** | 文档类型识别 → 角色选派 → 公开分级 | 角色名单+分级声明 | 协调方 |
| **1** | 隔离审阅 + 覆盖登记 + 签署声明 | 问题清单+覆盖声明 | 参审角色 |
| **2** | 同类合并 + 侧面互补 + 按章排序 | 归一化清单+对照表 | 协调方 |
| **3** | P0→P1→P2 逐项编辑 + 交叉验证 | 修复后文档+记录表 | 编辑方 |
| **4** | P0 逐条闭合 + P1 自检抽查 + P2 自检 | 闭合清单+审查档案 | 原角色+编辑方 |

> **与 SA-DM 治理闭环的关系**：文档类型识别=感知，角色选派=分流，审查+修复=适配，复审归档=反馈。审查严重度分级构成 dim10，审查档案复用 V2/V3 防篡改机制。
