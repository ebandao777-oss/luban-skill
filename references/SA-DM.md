# Agent智能体技能进化方法论

## Scenario-Adaptive Dual-Mode Architecture (SA-DM)

### 核心法则

- **"有道之魂.有术之能"**

- **【单模型“幻觉”风险】** 单模型处理复杂任务易产生幻觉，源于概率预测本质与知识边界盲区。长链路推理中单步错误/漂移会累积放大。必须通过多角色协同与S级模型全局视野捕捉隐性错误。

- **【评审架构阶梯式能力】** S级模型多角色协同评审 > A级模型多角色 > 单模型多角色 > 单模型单角色。

- **【基座“高杠杆”优化】** 技能专项优化采用顶级 S 级模型（Claude / GPT-5 / Qwen3.7-Max / DeepSeek-V4 Pro）作为基座打磨，是获取最高 ROI 的唯一解。

## 1. EvoSkill — 失败驱动的技能修补

- **地址**：[arXiv:2603.02766](https://arxiv.org/abs/2603.02766)
- **简介**：三 Agent 协同，从失败中自动发现能力缺口，构建结构化技能文件夹（含元数据/指令/脚本），产出可迁移的技能资产。
- **落地思路**：skill 执行报错时自动抓取失败上下文 → 分析缺口类型（规则缺失/触发词遗漏/指令冲突）→ 输出修改建议。

## 2. SkillOps — 定期健康巡检

- **地址**：[arXiv:2605.13716](https://arxiv.org/abs/2605.13716)
- **简介**：将技能库维护形式化为「库时」问题。五维健康诊断：效用/冗余/兼容性/失败风险/验证缺口。自动执行 merge/repair/retire/add_validator。
- **落地思路**：定时巡检 references 引用断裂、SKILL.md 规则冲突、未测试规则、可合并去重项。

## 3. CASCADE — 领域知识自动补充

- **地址**：[arXiv:2512.23880](https://arxiv.org/abs/2512.23880)
- **简介**：两大元技能驱动——持续学习（搜索/提取/记忆）和自我反思（内省/知识图谱），技能以可执行代码跨智能体共享。
- **落地思路**：references 外部知识过时时，自动搜索+验证 → 更新到 references/。

## 4. Skill Distill — 判断何时该精简

- **地址**：[arXiv:2604.01608](https://arxiv.org/abs/2604.01608)
- **简介**：首个可先验计算的技能效用预测器「指标自由度 F」，通过 Mantel 检验量化输出多样性与得分方差耦合。高 F 可精简，低 F（≤0.6）需保留。
- **落地思路**：超阈值时计算各模块 F 值 → 高 F 标可合并/删除，低 F 保留。

## 5. HASP — 将建议规则硬化

- **地址**：[arXiv:2605.17734](https://arxiv.org/abs/2605.17734)
- **简介**：技能升格为可执行程序函数（PF），含 should_activate（触发）和 intervene（干预）两接口，从"建议"变"硬纠正"。
- **落地思路**：分析执行日志定位被忽略的规则 → 加触发条件+硬执行描述 → 改为`强制：当 X 发生时必须执行 Y`格式。

## 6. MUSE-Autoskill — 修改后自动回归测试

- **地址**：[arXiv:2605.27366](https://arxiv.org/abs/2605.27366)
- **简介**：技能全生命周期框架，双驱动评估（单元测试+运行反馈），自动触发修补和重测。首次实证跨智能体技能迁移。
- **落地思路**：编辑前自动生成测试用例（触发词/格式/规则遵守）→ 修改后跑回归 → 退化回滚。

## 核心闭环：EvoSkill + SkillOps 双轮驱动

```
技能执行 → 出错
    ↓
EvoSkill 分析失败 → 生成补丁
    ↓
MUSE-Autoskill 验证补丁
    ↓
SkillOps 定期体检防退化
    ↓
CASCADE 补领域新知识
    ↓
HASP 硬化关键规则
    ↓
Skill Distill 精简冗余
```

**适用**：tiangong-skill / luban-skill / prompt-optimizer 等任何已有 skill。

## 二、各模块详细设计

### 模块 1：EvoSkill —— 失败驱动的技能修补

**触发条件**：用户反馈技能不符合预期，或 Agent 遇到明确能力缺口。

**执行流程**：

```
Step 1: 捕获失败上下文（原始指令 / 触发参数 / 输出报错 / 用户反馈）
Step 2: 定位缺口类型

| 缺口类型 | 判定特征 |
|----------|----------|
| 触发词遗漏 | 含关键意图但技能未触发 |
| 规则缺失/模糊 | 未覆盖用户场景 |
| 指令冲突 | 多条规则矛盾 |
| 流程漏洞 | 某环节不可行 |
| 输出格式不当 | 输出不符合预期 |
| 版本兼容 | 外部工具版本变更 |

Step 3: 生成补丁（定位位置 + old_str→new_str + 修改理由）
Step 4: 输出确认卡片 → 待用户确认后执行
```

**输出格式**：Markdown 修复建议卡片（含目标技能/缺口类型/定位/当前→建议/理由）。

---

### 模块 2：SkillOps —— 定期健康巡检

**触发条件**：定时（建议每周）或手动触发"检查技能健康"。

**五维诊断矩阵**：

| 维度 | 检查项 | 方法 | 严重程度 |
|------|--------|------|----------|
| **效用** | 每条规则是否有触发条件 | 扫描 trigger 覆盖率 | 🟡 |
| | references 是否被 SKILL.md 引用 | 交叉比对引用链 | 🟢 |
| **冗余** | 内容高度重复的 references | 文件相似度（Jaccard） | 🟢 |
| | SKILL.md 可合并的重复描述 | 段落语义去重 | 🟢 |
| **兼容性** | 文件路径引用断裂 | 逐条路径可访问性检查 | 🔴 |
| | 外部链接失效 | HTTP HEAD 检查 | 🟡 |
| | YAML frontmatter 合法性 | YAML 解析校验 | 🔴 |
| **失败风险** | 未定义触发条件的强制性规则 | 扫描"必须"/"禁止" | 🟡 |
| | 规则粒度过粗导致歧义 | 统计约束项数量 | 🟢 |
| **验证缺口** | 缺少验证步骤 | "验证"关键词覆盖 | 🟡 |
| | references 缺少示例/反例 | 扫描 example/counterexample | 🟢 |

**执行流程**：

```
Step 1: 加载 SKILL.md + references/
Step 2: 逐维度扫描（工具化：路径/YAML/引用链；语义化：一致性/重复/歧义）
Step 3: 汇总诊断报告（按严重度排序：🔴断裂 > 🟡风险 > 🟢优化）
Step 4: 生成维护动作清单（merge/repair/retire/add_validator）
```

**维护动作定义**（对齐 SkillOps 论文概念）：

| 动作 | 含义 | 示例 |
|------|------|------|
| `merge` | 合并内容高度重复的 references | ref-a.md 和 ref-b.md 语义相似度 > 0.8 → 合并 |
| `repair` | 修复断裂引用或非法格式 | 路径 `/old/path.md` 不存在 → 改为 `/new/path.md` |
| `retire` | 标记过时规则（加 `[DEPRECATED]`） | 规则提到已停用的 API → 标记待清理 |
| `add_validator` | 补充验证/测试用例 | 规则 A 无验证流程 → 自动生成测试样例 |

### 模块 3：CASCADE —— 领域知识自动补充

**触发条件**：references 引用外部知识且距上次更新 > 90 天。

**执行流程**：

```
Step 1: 扫描 references/ → 识别外部引用（arXiv ID / URL / 标准编号）
Step 2: 筛选过时引用（> 阈值，优先高频技能）
Step 3: 知识检索（搜索论文/抓取 API/查标准新版本）
Step 4: 自我反思 → 仅在有实质性变化时更新
Step 5: 追加式更新（不删旧内容，标注 [YYYY-MM-DD]）
```

**关键设计**：只追加不替换；规则废弃留给 SkillOps `retire`。

---

### 模块 4：Skill Distill —— 判断何时该精简

**触发条件**：SkillOps「冗余」维度评分低，或 references 文件数 > 15。

**F_approx 近似**：

```
F_approx = 1 - (模块被规则引用次数 / 模块总字符数归一化)
```
- F ≥ 0.7：可精简；≤ 0.3：核心资产；0.3<F<0.7：暂不处理

**执行流程**：

```
Step 1: 构建引用矩阵 → 计算有效引用密度
Step 2: 计算 F_approx 分级标记
Step 3: 生成精简建议（提取核心段落/删未引用部分/提升高密度段落到 SKILL.md）
Step 4: 输出精简方案（大小变化预估+核心内容标注）→ 用户确认
```

**精简优先级**：

| 优先级 | 类型 | 典型处理 |
|--------|------|----------|
| P0 | 完全未被引用的 references 文件 | 直接建议删除 |
| P1 | 文件大但仅 1-2 处引用 | 提取引用段落到 SKILL.md，删原文件 |
| P2 | 多处重复的示例代码块 | 合并为一个 reference |
| P3 | 历史版本累积的旧内容 | 归档到 archive/ 子目录 |

### 模块 5：HASP —— 将建议规则硬化

**触发条件**：同一场景连续 ≥2 次未遵守关键规则。

**硬化层级**：

#### 层级 1：Should → Must（措辞强化）

```
原文：建议在生成 SKILL.md 时控制文件大小在 30KB 以内
硬化后：强制约束：SKILL.md 文件大小不得超过 30KB。超限时必须拆分到 references/。
```

#### 层级 2：Should → PF（可执行程序函数）

```yaml
hard_rules:
  - id: rule_001
    should_activate: "SKILL.md 文件大小 > 30KB"
    intervene:
      type: "block_and_restructure"
      action: "禁止继续追加内容，将超出部分写入新 reference 文件"
    severity: "critical"
    last_violated: "2026-06-10"
    violation_count: 3
```

**执行流程**：

```
Step 1: 执行日志分析 → 对比关键规则 vs 实际行为 → 识别被忽略实例
Step 2: 分级：1次=不处理，2次=措辞强化（层级1），≥3次=PF硬化（层级2）
Step 3: 硬化规则生成（提取约束→定义should_activate+intervene→注入hard_rules）
Step 4: 输出对比建议 → 用户确认
```

**硬化适用性判断**：

| 规则类型 | 是否适合硬化 | 原因 |
|----------|-------------|------|
| 输出格式约束 | ✅ 适合 | 可精确检测和强制修正 |
| 文件大小限制 | ✅ 适合 | 可精确检测 |
| 必须包含的章节/字段 | ✅ 适合 | 结构化检查 |
| 语义风格约束 | ❌ 不适合 | 难以精确检测 |
| 创造性建议 | ❌ 不适合 | 无法形式化 |

### 模块 6：MUSE-Autoskill —— 修改后自动回归测试

**触发条件**：对技能文件的编辑操作完成后。

**测试用例生成规则**：

| 测试维度 | 生成方法 | 示例 |
|----------|----------|------|
| **触发词识别** | 从 SKILL.md 提取所有触发词，逐一构造输入 | 输入"小鲁班" → 预期触发 tiangong-skill |
| **输出格式** | 从 SKILL.md 提取格式约束，构造验收条件 | 输出必须包含 YAML frontmatter |
| **关键规则遵守** | 提取"必须"/"禁止"语句，构造边界测试 | 输入导致超限的请求 → 预期技能拒绝 |
| **流程完整性** | 按技能的 Step 列表逐项模拟 | Step 3 依赖 Step 2 的输出 → 断链测试 |
| **references 可达性** | 遍历所有文件路径引用 | 逐条检查文件是否存在 |
| **反例测试** | 构造明确不在技能范围内的输入 | "帮我写个操作系统" → 预期不触发技能 |

**执行流程**：

```
Step 1: 修改前快照 → 保存 hash + 生成 5-10 条测试用例（6 维度）
Step 2: 执行修改
Step 3: 回归测试 → 逐条检查修改后行为
Step 4: 结果判定（全部通过→通过；部分失败→列偏差+建议回滚；全部失败→强制回滚）
Step 5: 通过的用例追加到 tests.yaml
```

**tests.yaml 格式**：

```yaml
skill: tiangong-skill
generated_at: 2026-06-11
tests:
  - id: trigger_001
    dimension: "触发词识别"
    input: "小鲁班帮我优化下这个skill"
    expected: "触发 tiangong-skill"
    status: pass
    last_run: 2026-06-11
  - id: format_001
    dimension: "输出格式"
    condition: "SKILL.md 含 description 字段"
    check: "YAML 解析通过"
    status: pass
    last_run: 2026-06-11
  - id: rule_001
    dimension: "关键规则遵守"
    input: "不分析直接修改文件"
    expected: "应拒绝，要求先分析"
    status: fail
    fail_reason: "未拦截直接修改"
    last_run: 2026-06-11
```

## 三、调度策略

### 3.1 触发优先级

```
事件驱动（立即）
  ├── P0: 用户反馈技能错误 → EvoSkill
  ├── P0: 技能编辑完成 → MUSE
  ├── P1: 规则忽略 3 次 → HASP 硬化
  └── P2: 规则忽略 2 次 → HASP 措辞强化

定时驱动（周期）
  ├── 每周：SkillOps 巡检
  ├── 每月：Distill 精简检查（冗余评分低时）
  └── 每季度：CASCADE 知识更新（有外部引用时）
```

### 3.2 并发控制

- 同技能同时只允许一个演化任务
- 新任务 FIFO 排队
- MUSE 运行期间锁定技能文件

### 3.3 冲突仲裁

| 场景 | 规则 |
|------|------|
| EvoSkill vs 用户手动编辑 | 用户优先 |
| SkillOps vs 用户编辑 | 只读执行 |
| Distill vs HASP | HASP 优先 |

## 五、与调度器集成

独立技能 `skill-evolution`，由 Marvis dispatch_task 调度执行，返回分析报告/修改建议/修复结果。

**调度判断表**：

| 用户指令关键词 | 触发模块 | 是否需要用户确认 |
|---------------|----------|-----------------|
| "检查技能"、"技能体检" | SkillOps | 否（仅输出报告） |
| "这个技能有问题"、"技能出错了" | EvoSkill | 是（修改需确认） |
| "更新技能知识"、"补最新" | CASCADE | 否（仅追加不修改） |
| "精简技能"、"技能太长了" | Skill Distill | 是（删除需确认） |
| "这个规则总被忽略"、"硬一点" | HASP | 是（修改需确认） |
| "改完后测试下"、"不会退化吧" | MUSE | 否（自动跑） |
